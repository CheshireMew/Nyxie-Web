# 互动与视频导演实现

用于把多段独立生成视频组织成自主表演、偶然互动和滚动剧情，并避免抢播、卡顿与状态错乱。

## 0. 编码前冻结路由与状态所有权

在创建事件监听器、播放器 hook 或时间线前，先写完：

```text
有序章节唯一真源：
唯一导航入口：
每个互动入口：起点 / 片段 / 目标章节 / 对齐方式
正常启动链路：HTML 首帧 / 应用加载界面 / 正常首页 / 故障降级及其触发条件
媒体职责：文件 / playbackRate / muted / 声音归属 / preload / 完成策略
媒体资源时序：冷启动预算 / 预热触发 / 目标身份 / 可用证据 / 激活行为
完整页面加载语义：首屏剧情 / 欢迎回应 / Gallery 样本 / 刷新位置
Gallery：完整目录 / 本次样本 / 当前形态 / 各入口可访问范围
交互命中：可见卡片 / 实际链接 / 弹层 / 遮罩 / 底层不可触发区域
无动画备用结果：
播放中途失败结果：
离散章节状态生产者：
连续进度唯一所有者：
最终稳定状态断言：
```

成功路径、无动画备用和减少动态效果共享同一互动路由；播放中途失败按失败合同恢复。按钮、滚轮和媒体回调只提交请求，不各自决定视频源、章节 id 或滚动方式。上述字段没有具体值时不开始编码，不能先写三个硬编码调用再等集成测试发现分叉。

## 1. 先建立唯一状态边界

页面只能有一个动画导演。按钮、滚轮、空白点击、自动时间线和返回首屏都调用导演公开方法。

导演至少维护：

```text
activePlayer       当前可见播放器
standbyPlayer      备用播放器
currentKey         当前片段
transitioning      是否正在切换
requestPriority    当前请求优先级
exclusivePending   已受理但尚未完成切换的独占请求
lockedUntilEnd     当前片段是否独占到真实片尾与导航稳定
plannedKey         已计划的下一片段
completionPolicy   当前片段的 loop / hold / replay / advance-on-ended
visible            首屏是否可见
documentVisible    页面是否在前台
storyConsumed      本次完整页面加载是否已消费首屏剧情
after              当前动作完成后的任务
```

不得在多个事件监听器里分别调用 `video.src = ...`、`play()`、`pause()` 或切换可见类。那会产生多个状态真源。

## 2. 给请求分优先级

推荐顺序：

| 优先级 | 类型 | 示例 | 规则 |
| --- | --- | --- | --- |
| 3 | 叙事锁定 | 首次滚动开门 | 播放完成前不接受其它片段 |
| 2 | 明确操作 | 动作按钮、对话入口 | 可替换普通待机，不打断叙事锁定 |
| 1 | 偶然互动 | 空白点击、返回回应 | 导演空闲时执行 |
| 0 | 自动表演 | 待机、随机动作 | 随时可被更高优先级替换 |

切换正在进行时，新请求应排队、合并或拒绝，不能同时启动第二次切换。连续相同请求只保留一次。

### 明确操作不能被自动回应吞掉

采用以下仲裁顺序：

1. 叙事请求一经受理便建立独占锁；等待导演就绪、等待当前转场和加载备用播放器期间也不接受其它片段、导航或重复输入。
2. 普通转场进行时，新请求先记录，不启动第二个转场。
3. 多个待处理请求只保留最高优先级；同优先级保留用户最后一次明确操作。
4. 当前转场结束后重新检查页面和播放器状态，再执行保留请求。
5. 自动待机、返回欢迎和随机动作不能覆盖已经排队的按钮、目录或对话请求。

典型竞争：用户返回首屏后，欢迎回应刚开始切换，又点击指定动作。导演应让当前转场完成，随后执行用户最后点击的动作；不能因为 `transitioning === true` 直接丢弃该点击，也不能让欢迎回应持续占用状态。

推荐额外维护：

```text
requestSequence   递增请求编号
pendingRequest    当前待处理请求
pendingPriority   待处理请求优先级
storyLocked       是否处于不可抢占叙事
```

请求完成、被更高优先级替换或状态失效后，明确清除待处理项，不能留下跨章节的陈旧动作。

## 3. 使用双视频缓冲

两个 `<video>` 层交替承担活动和备用：

1. 活动播放器正常播放。
2. 导演根据当前片段提前决定下一片段。
3. 备用播放器设置 `src`，调用 `load()`，等待 `loadeddata`。
4. 切换时不再等待网络下载，直接从备用播放器启动。
5. 切换成功后交换活动索引，再给新的备用播放器加载后续片段。

每次加载和转场使用递增的 `operationId`，并捕获本次拥有的播放器节点、目标 key 与目标 `src`。`load()`、`play()`、`loadeddata`、`canplay`、`ended` 等异步边界返回后，都要确认操作编号、节点、`currentSrc` 和目标 key 仍属于本次操作；过期操作直接作废。旧操作的清理只能处理它捕获且仍处于旧角色的节点，不能通过可变的“当前播放器”引用暂停、归零或隐藏已经被新操作接管的节点。

播放前统一设置：

```text
currentTime = 0
playbackRate = 当前请求值
defaultPlaybackRate = 当前请求值
muted = 媒体清单中的该职责值
```

速度与静音都从媒体清单读取，并在每次交换播放器、重播或错误恢复后重新应用。人物视频与独立 BGM 是不同声音通道；如果合同规定人物视频始终静音，任何全局声音按钮都不能修改它们。

`play()` 必须等待 Promise。带声音自动播放失败时，切回静音再试；再次失败则恢复旧画面或锚点。

### 预热与播放是两个阶段

预热必须从正式媒体清单和当前会话状态读取目标，不能复制一份测试专用 URL，也不能让“本次样本”和“当前形态”分别决定不同文件。对 Gallery 这类即将到达的章节，合同可以规定在首屏开门请求被受理后开始预热：至少请求本次会话选中的首个人物视频，以及进入时真实可见的当前与相邻海报。冷启动阶段仍只加载首屏预算内资源，避免为了预热整个目录延迟首页。

预热播放器只设置正确的 `src`、`preload` 并调用 `load()`，保持暂停且不消费播放进度。图片可用要由目标元素的 `complete`、自然尺寸和 `decode()` 证明；视频可用要由 `currentSrc` 对应目标、`readyState >= 2` 或该目标的 `loadeddata/canplay` 证明。空资源清单表示“本次没有预热任务”，不是“Gallery 已准备好”。

章节激活时重新校验目标身份，按媒体清单恢复 `playbackRate`、`defaultPlaybackRate`、`muted` 和完成策略，再把 `currentTime` 设为 0 后调用 `play()`。预热失败不得阻断导航，但到达时要显示同一形态的已解码海报或锚点，不能先暴露空舞台再等待另一个异步请求。

### 完成策略来自媒体合同

每个片段明确使用以下一种策略：

- `loop`：同一片段循环；只有真实素材首尾与浏览器循环效果通过验收时才宣称无缝。
- `hold`：停在合同允许的尾帧或对应海报。
- `replay`：通过同一导演边界从头重播，不在多个 `ended` 监听器里各自改 `currentTime`。
- `advance-on-ended`：真实 `ended` 后调用与滚轮、按钮、键盘和侧卡相同的 `selectCatalogItem()` 或导演选择边界切到下一人物。

把 `loop` 属性取消并不自动得到顺滑轮播；最终效果取决于片尾到下一人物的素材关系、预热是否完成以及隐藏备用层能否原子晋升。用户修改完成策略时，同步迁移媒体清单、监听器、旧恢复逻辑、测试和维护样本，不保留仍会触发旧循环或重播的第二条路径。

## 4. 自动表演不是静态图夹视频

不使用以下循环：

```text
锚点图 → 待机视频 → 锚点图 → 待机视频
```

锚点图只在首次加载、切换模糊峰值和错误回退时出现。正常时间线直接在视频片段之间切换：

```text
主待机 → 次待机 → 随机动作 → 主待机
```

随机动作排除上一次动作。动作结束后回到主待机，避免多个强动作连续出现。

普通自动序列可以在片尾前约 0.35–0.65 秒开始切换，具体值由动作结尾决定。叙事片段不能提前切换。

## 5. 失焦锚点切换

推荐用 Web Animations API 操作整个媒体舞台的 `filter`，而不是让视频层分别淡化。

伪代码：

```js
async function transition(previous, next) {
  await ensureLoaded(next);

  await heroMedia.animate(
    [
      { filter: "blur(0px) brightness(1)" },
      { filter: "blur(14px) brightness(1.02)" },
    ],
    { duration: 180, easing: "cubic-bezier(.4,0,1,1)", fill: "both" }
  ).finished;

  previous.pause();
  previous.classList.remove("is-visible");

  const started = await startAtZero(next);
  if (!started) return restorePrevious(previous);
  next.classList.add("is-visible");

  await heroMedia.animate(
    [
      { filter: "blur(14px) brightness(1.02)" },
      { filter: "blur(0px) brightness(1)" },
    ],
    { duration: 260, easing: "cubic-bezier(.22,1,.36,1)", fill: "both" }
  ).finished;
}
```

通过标准：

- 模糊只影响角色媒体，不影响导航、标题和按钮。
- 切点发生在模糊峰值。
- 任意时刻最多一个人物视频可见。
- 锚点不会清晰停留形成“卡住一帧”的感觉。
- 新视频失败后能恢复可用画面和正确导演状态。

## 6. 偶然互动的事件边界

空白或角色点击可以触发回应，但先排除：

```text
a, button, input, textarea, select, [contenteditable]
首屏文案区
动画控制区
滚动提示
打开的弹窗和面板
```

随后确认：

- 首屏仍可见。
- 页面位于首屏顶部容差范围内。
- 点击发生在首屏内部。
- 导演没有叙事锁定。
- 当前没有需要优先完成的明确操作。

按钮触发后立即 `return`，避免同一次冒泡事件又触发空白点击回应。

## 7. 第一次滚动的章节门控

滚动剧情是输入门控，不是普通滚动监听。

状态：

```text
storyConsumed  本次完整页面加载是否已经受理过首次剧情
storyLocked     是否正在播放剧情
touchStartY     触摸起点
```

“每次真正初次加载或整页刷新只播放一次”属于完整页面加载生命周期，不属于滚动位置：

- 首次有效向下输入被受理时立即设置 `storyConsumed`，不能等播放成功后才设置。
- 通过站内导航或原生滚动回到首页、回到顶部、首屏组件重新进入视口，都不清除它。
- 该状态放在本次文档生命周期稳定存在的应用级 ref 或 store，不放进会因路由或章节重挂载而重置的局部组件，也不用 `sessionStorage` 冒充“整页刷新后重置”。
- 整页刷新创建新的文档状态；如果合同还要求刷新回到顶部，在框架脚本运行前设置 `history.scrollRestoration = "manual"` 并清零滚动，不能等 React 挂载后再与浏览器的滚动恢复竞速。
- 自动剧情失败后保持“已消费”，页面留在首屏并提供显式重试；重试是用户明确操作，不让普通滚轮静默反复启动失败剧情。
- `storyConsumed` 与 Gallery 预热状态彼此独立。合同要求用开门过程提供预热窗口时，首次请求一经受理便提交 Gallery 预热；预热失败不重置 `storyConsumed`，也不让普通滚轮再次播放开门来“补加载”。

页面的有序章节和互动路由是导航目标的唯一来源：

```js
const storyRoute = {
  clip: "story",
  origin: "home",
  destination: getNextSection("home"),
  alignment: "start",
};
```

`getNextSection()` 读取与 DOM、主导航和目录相同的章节清单。表示“下一章节”的滚轮、滚动提示、体验合同允许的无动画备用路径和减少动态效果路径都使用 `storyRoute.destination`；播放中途的意外失败仍留在 `origin` 并提供重试。明确写着“读取角色档案”等允许跳章的业务按钮单独建路由，不能充当首次滚动的备用目标。

流程伪代码：

```js
async function beginStoryScroll() {
  if (!canStartStory()) return false;

  storyConsumed = true;
  storyLocked = true;
  try {
    await director.waitUntilReady();
    const started = await director.request(storyRoute.clip, {
      priority: 3,
      lockUntilEnd: true,
    });
    if (!started) throw new Error("story video could not start");

    const ended = await director.waitUntilEnded(storyRoute.clip);
    if (!ended) throw new Error("story video did not finish");

    await navigation.goAndWait(storyRoute.destination, storyRoute.alignment);
    return true;
  } catch (error) {
    showRetryOrStaticFallback(error);
    return false;
  } finally {
    storyLocked = false;
  }
}
```

导演的 `request()` 从接收 `lockUntilEnd` 请求开始设置 `exclusivePending`，并用 `return await transitionAndPlay()` 保持锁直到异步切换真正完成；不能在返回内部 Promise 后过早清除 pending。当前锁定片段、待处理独占请求和导航锁共同组成公开的 `isInteractionLocked()`。导航、动作按钮、随机播放、对话入口、空白点击、键盘和滚轮都在控制器边界查询它，不能只在各自 UI 上零散禁用。

滚轮监听必须使用 `{ passive: false }`。当 `storyLocked` 为真时，每一次滚轮都 `preventDefault()`。触摸端在 `touchstart` 记录起点，在向上移动超过阈值后触发，并在锁定期间阻止 `touchmove`。

`waitUntilEnded` 同时监听 `ended` 并检查：

- 当前播放器仍是启动剧情时的播放器。
- 当前片段仍是剧情片段。
- `player.ended === true`。

超时只用于释放死锁和显示失败，不滚动到下一章节。

`navigation.goAndWait()` 是页面唯一导航边界：它处理关闭弹层、滚动方式、固定导航留白和活动章节消费，并在目标章节达到互动合同约定的几何锚点且滚动值连续稳定后完成。默认 `alignment: "start"` 时，目标顶部应与 `scroll-padding` 对齐；其它对齐方式必须在路由中明确记录。支持 `scrollend` 时可以监听该事件；仍要核对目标几何位置，兼容路径可连续采样多个动画帧确认 `scrollY` 和目标 `getBoundingClientRect()` 不再变化。

不能在动画完成回调里直接查询某个 DOM id 并 `scrollIntoView()`，同时让按钮、目录和减少动态效果路径各自维护另一套目标。DOM 顺序、导航标签或 `IntersectionObserver` 在平滑滚动途中短暂显示目标章节，都不能证明最终去向正确。

## 8. 后续章节先选滚动架构，再建立生命周期

首屏故事门控完成，不等于所有后续章节只能使用同一种滚动办法。先按用户实际要完成的动作分类，再为每类章节指定唯一输入和进度所有者：

| 章节类型 | 默认架构 | 输入所有者 | 适用边界 |
| --- | --- | --- | --- |
| 被动内容 | 原生文档流 | 浏览器 | 阅读、链接和普通内容区 |
| 章内分段演出 | 原生滚动 + sticky 舞台 | 浏览器提供连续位置，章节导演只消费位置 | Character、Personality 等连续构图推进 |
| 全屏内容换场 | 固定全屏舞台 + 内容状态切换 | 浏览器提供连续位置，换场导演只消费位置 | 用户同时要求无强制吸附感且换场时绝不出现两页拼接 |
| 显式局部步进 | 完整就位后的有限输入门 | 局部组件只消费尚未完成的向下步进 | Gallery 形态轮播等用户明确可逐项查看的交互 |
| 分页落页 | 分页吸附 | 页面导演 | 只有用户明确接受离散落页手感时 |

普通文档流可以自然、跟手，但页面交界会按真实几何短暂出现；分页吸附能保证完整落页，却会改变触控板与反向滚动手感。用户同时要求“不要吸附”和“任何时刻都不能看到两页交界”时，选择固定全屏舞台换场，不能把两种要求塞进普通文档流后反复调整阈值。

每个有章内演出的章节都使用同一生命周期真源：

| 阶段 | 几何与可见结果 | 章内进度 | 下一页 | 反向 |
| --- | --- | --- | --- | --- |
| 进入构图 | 章节从视口外进入，舞台尚未到合同锚点 | 固定为 0 | 不参与章内演出 | 原生返回 |
| 完整就位 | 舞台完整占据表演视口，标题、人物和关键编号均无遮挡 | 仍为 0 | 不可进入视口 | 保持当前稳定画面 |
| 章内演出 | sticky 或固定舞台保持完整，状态按滚动位置推进 | 从 0 单调到 1 | 不可进入视口 | 不降低已达到的可见状态 |
| 终态释放 | 所有必须出现的信息已稳定在终态 | 固定为 1 | 此时才可进入视口 | 立即越过视觉不再变化的已消费距离 |

进入进度和章内进度必须由不同边界计算。章节顶部尚未到合同锚点时，无论页面已露出多少，章内进度都为 0。章内演出开始后，连续视觉值直接映射实际滚动位置；默认不用带延迟的时间动画追赶滚动，也不用平滑 scrub 制造“高级感”。离散状态只在跨过阈值时更新一次。

章节完成后记录最高已完成进度。反向返回只恢复该章的稳定终态，不倒放、不重新入场。为了避免用户先滚过一段视觉完全不变的占位距离，可以在完成后折叠不再需要的演出跑道并补偿当前视口位置，或在反向进入已消费跑道时一次性转移到它的起始边界；这次转移必须由第一次有效反向输入触发并立即产生可见位移，不能要求连续滚轮多次。

章节高度按有意义的可见状态估算：

| 章节复杂度 | 初始屏数建议 | 使用条件 |
| --- | --- | --- |
| 单一构图变化或两态切换 | 约 1.4–2.0 屏 | 开始与结束都能直接理解 |
| 三到四个有信息增量的阶段 | 约 2.2–3.0 屏 | 每个阶段都有新证据、关系或动作 |
| 超过 3 屏 | 单独说明 | 必须存在不能合并的新状态，不能只是停留或等待 |

这只是初始调校范围。真正标准是：完整就位前没有章内演出，演出期间下一页不可见，下一页首次出现时当前章已经是终态，正向每段距离都有推进，反向第一下就有反馈。

### Gallery 的局部步进合同

Gallery 不用一个数组同时承担“全部可用内容”和“本轮滚轮序列”。至少维护：

```text
catalog          媒体清单中的完整形态目录
sampledIds       本次完整页面加载随机抽取并保持稳定的有限样本
activeCatalogId  当前活动形态，始终属于完整目录
viewedSampleIds  本轮滚轮已经推进的样本
```

样本数量来自体验合同；三个只是常见项目值。组件只有在完整就位后才处理局部滚轮：

- 向下滚动时，如果还有本轮未展示形态，一次有效手势只切换一个，并短暂抑制同一手势的惯性尾部。
- 样本全部展示后，下一次向下滚动不再拦截，直接进入下一章。
- 向上滚动始终交还浏览器，不要求按相反顺序消费完剩余样本。
- 局部滚轮按 `sampledIds` 推进；左右按钮、方向键和侧卡按合同遍历 `catalog`，因此可以加载样本之外的形态。
- 所有入口调用同一个 `selectCatalogItem(id, source)` 边界，共用转场、焦点、节流、视频速度、静音和状态提交；样本位置只由滚轮入口额外维护。
- 首次开门过程若是合同中的预热窗口，预热目标必须来自同一 `sampledIds` 与 `activeCatalogId` 真源；至少准备进入时的活动视频以及当前和相邻可见海报，不在到达 Gallery 后才临时生成另一组选择。
- 用户通过按钮、方向键、侧卡或再次进入 Gallery 返回任意形态时，按该媒体的完成策略从头播放或恢复；不得用“回收旧视频”的清理回调重置或重播当前节点。
- 活动视频是否播放由 Gallery 的真实可见性和当前形态决定；向上滚动方向本身不是暂停条件。章节重新进入视口时恢复当前视频，完全离开后才暂停。
- 切换期间新人物在隐藏备用层加载和首帧就绪；晋升活动身份、可见层级和 `activeCatalogId` 在一次提交中完成，随后才回收确认仍为旧身份的节点。任一稳定帧最多有一个人物层可见，不能用两层交叉叠加掩盖素材不对齐。
- 切换期间视觉状态可以短暂锁定，但不能锁住页面滚动；新请求按同一导演边界合并或排队。异步等待只接受期望的形态 id、节点和 `currentSrc`，不能被“任意 active video 已 ready”提前满足。
- `ended` 后是否停留、重播、循环或自动切到下一人物由当前条目的完成策略决定。`advance-on-ended` 仍调用 `selectCatalogItem(nextId, "ended")`，因此和滚轮、按钮、键盘、侧卡共享同一预热、转场与去重边界。

只有这个明确的局部交互可以有限消费滚轮。Character、Personality、Works 等普通章节不得复制 Gallery 的监听器，也不得使用全局滚轮倍率、持续 `preventDefault()` 或强制翻页掩盖章节过长。

## 9. 独占请求与自动推进

普通片段可以在临近片尾时调用 `advance()`。设置 `lockUntilEnd` 后：

- 请求等待转场和加载时 `exclusivePending` 已经为真。
- `timeupdate` 不触发自动推进。
- `ended` 不切回待机。
- 页面重新可见时不重新播放已经结束的片段。
- 真实 `ended` 后仍由叙事流程完成导航；目标稳定或失败恢复完成后才解除独占状态。

锁只在导演和导航控制器中维护。UI 可以显示禁用状态，但不能成为唯一防线；否则延迟按钮、全局导航或程序化请求仍可能打断叙事。

## 10. 离开和返回首屏

用 `IntersectionObserver` 观察首屏：

- 离开首屏时暂停活动播放器，记录用户确实离开过。
- 返回首屏时先恢复可用画面，再按体验合同触发一次欢迎回应。
- 因“下方动作卡片要求返回首屏并播放指定动作”而返回时，抑制自动欢迎，避免两个动作抢播。
- 用户在欢迎回应切换期间选择了明确动作时，按请求优先级排队执行用户动作；不能为了欢迎效果忽略用户输入。
- 页面进入后台时暂停；回到前台时只在首屏可见且没有片尾锁定时恢复。

## 11. 导航、媒体声音和减少动态效果

- 固定导航高度应成为媒体舞台的布局变量。媒体从导航下方开始，同时向底部补偿首屏高度，避免人物头部被遮挡。
- 为人物视频和独立 BGM 分别写声音合同。人物视频默认静音并保证自动播放；如果用户要求始终静音，声音按钮只启动、暂停独立 BGM，标签和 `aria-pressed` 也只反映 BGM。
- BGM 的默认状态采用用户最新决定。默认关闭时页面加载后保持暂停；用户开启后从约定起点播放并循环，关闭后暂停。音量、真实时长、来源区间和循环行为进入媒体清单与浏览器验收。
- 浏览器下载器扩展接管音频直链时，移除 `<audio>` 上的源文件 URL，由页面 `fetch` 原文件、验证响应、创建带正确 MIME 的 Blob URL 并交给播放器。组件卸载时中止请求、撤销 URL；播放失败仍走同一个重载与用户手势恢复边界。
- `prefers-reduced-motion` 下不启动自动导演，显示锚点，并通过同一互动路由直接进入目标章节；后续内容使用可读的静态布局，不保留没有信息价值的固定滚动距离。
- 粗指针设备不用悬停作为唯一入口；按钮和触摸手势都必须可用。

## 12. 弹层、链接、章节触发和自定义光标

可见范围与命中范围使用同一份几何合同：

- 链接元素只包住用户看见并理解为入口的卡片或文字。作品展示图、整章容器和弹层后方内容不因实现方便成为大面积 `<a>`。
- 弹层遮罩位于页面内容之上、弹层之下，显示时启用指针命中；弹层外点击由遮罩消费，不能穿透到底层链接。隐藏时同时取消可见性和命中，避免透明层阻塞页面。
- 自动出现的末章链接卡片由章节生命周期的稳定终态触发。滚轮监听可以观察输入，但触发条件不能依赖鼠标位于视口中间、某个子元素 hover 或特定横坐标。
- 固定导航高度是首屏媒体舞台和滚动布局共享的唯一变量；导航滚动样式只能改变视觉，不额外叠加 margin、padding 或占位间隙。
- 自定义光标外层只负责 `translate3d(pointerX, pointerY, 0)`，内部伪元素或子层负责 hover 放大。弹簧跟随可以保留，但每帧把圆心与真实指针的距离限制在合同半径内，并在停止后收敛到同一点；不能用独立 `scale` 放大同时带有平移的定位层。

这些边界不能只通过选择器和事件单测验收。真实浏览器在卡片内、卡片外、弹层外和不同横向位置发送坐标点击或滚轮，再检查 `elementFromPoint()`、新页面目标、当前 URL、章节状态和光标圆心。

## 13. 失败恢复

每个异步阶段都定义恢复结果：

| 失败位置 | 恢复动作 |
| --- | --- |
| 备用视频加载失败 | 保持当前视频或锚点，状态不切换 |
| 新视频 `play()` 失败 | 恢复旧视频，清除模糊和切换锁 |
| 叙事视频未结束 | 留在首屏、解除输入锁、显示重试 |
| 导航未稳定到目标章节 | 保留可用页面、解除输入锁，并提供使用同一路由目标的明确入口 |
| 页面切到后台 | 暂停，回到前台后按状态恢复 |
| 用户连续触发 | 按优先级拒绝、合并或排队 |

恢复逻辑属于导演，不在各个按钮或事件监听器里各写一套。
