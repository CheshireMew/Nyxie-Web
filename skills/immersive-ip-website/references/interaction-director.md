# 互动与媒体运行边界

用于把页面互动、媒体选择、资源预热和可选的滚动演出组织成可理解、可维护的运行链路。先选择当前页面真正需要的机制，不预设自主表演、双播放器、固定观察窗或首屏叙事。

## 1. 先写互动映射

每项互动在编码前写清：

| 输入 | 生效区域 | 读取什么 | 反馈出现在哪里 | 反馈内容 | 可打断性 | 触摸或键盘替代 | 结束状态 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| | | | | | | | |

输入位置、取样位置和反馈位置是三个独立决定。指针可以直接改变原位置，也可以只提供坐标，让另一个固定区域显示结果；两者都不是默认答案。只有反馈位置、大小和信息价值符合页面目标时，互动才进入实现。

可见几何和命中几何必须一致。真实链接只覆盖可见入口；遮罩需要阻止底层命中；自定义指针的定位层与缩放或装饰层分开。

## 2. 分开最终布局、动画位移和命中几何

含推入、弹出、放大、跟随、悬浮或居中定位的元素，先填写：

| 对象 | 最终布局锚点 | 动画起点 | 运动属性 | 命中区域 | 结束后的稳定几何 |
| --- | --- | --- | --- | --- | --- |
| | | | | | |

三层职责分别是：

- 布局层决定元素最终相对视口、容器、基线或相邻对象的位置；
- 运动层只表达从起点到终点的视觉变化；
- 命中层只负责可操作范围、遮罩拦截和关闭边界。

当居中或锚定本身依赖 `transform` 时，使用稳定外层负责布局，内层负责动画位移；或者改用不会与动画属性竞争的布局方式。动画库、CSS transition 和布局规则不能同时写同一最终变换。视觉上已经出现不等于到位，必须分别观察入场方向、运动中状态和动画结束后的精确坐标。

时间关系也要分开记录：启动延迟决定何时出现反馈，运动时长决定到达速度，缓动决定速度分布，阶段重叠决定总等待，媒体 `playbackRate` 决定成片播放。用户只要求某段动效略快或略慢时，先确定实际控制其感知区间的唯一时间源，只调整该变量并保留已确认的终点、位移、内容停留和其它阶段关系。

固定遮罩、定位外层和动画内层如果职责不同，可以嵌套存在；但真实链接和按钮只覆盖可见入口，遮罩需要阻止底层命中，动画结束后不能留下透明但可点击的旧层。

## 3. 选择最小运行结构

根据实际复杂度选择：

| 需求 | 结构 |
| --- | --- |
| 单段自动循环媒体 | 一个真实播放器、poster 和明确完成策略 |
| 多段媒体由多个入口选择 | 一份媒体清单和一个选择边界 |
| 切换时不能等待加载 | 当前消费者加可晋升的备用消费者，或框架支持的等价缓存边界 |
| 不可打断的叙事或导航过程 | 统一请求状态和明确独占范围 |
| 连续滚动驱动视觉 | 一个连续进度所有者 |
| 离散目录或形态选择 | 完整目录、当前选择和可选会话样本分别建模 |

不要为了“角色应该活着”自动加入待机序列、随机动作和点击彩蛋。它们只有在体验合同说明触发、反馈和结束状态后才成立。

## 4. 建立唯一媒体清单

正式页面、预热逻辑、错误回退和验收共同读取同一份清单：

```text
id
src 与 MIME
poster
页面职责
playbackRate
muted 与声音归属
preload
起播位置
完成策略
循环或转场边界
错误回退
```

媒体参数不能散落在组件、事件监听器和测试中。用户改变某一职责后，一次性迁移所有消费者并移除旧常量和旧监听。

完成策略可为循环、停留、重播、播放结束后选择下一项或由业务回调接管。策略只从当前体验和真实成片决定；取消 `loop`、增加淡化或换成另一格式都不会自动得到顺滑结果。

## 5. 预加载最终消费者

预加载目标由当前选择状态和正式媒体清单产生。完整链路是：

```text
用户仍在前一页面或前一状态
→ 正式选择逻辑确定即将显示的媒体
→ 最终播放器或可原子晋升的备用播放器设置目标
→ 请求和解码发生
→ 页面到达
→ 同一节点继续显示或被原子晋升为可见消费者
```

把三个状态分开：

```text
warmupRequested   已进入真实提前量，应给最终消费者挂载目标源并解码
sectionVisible    内容的像素可能进入视口，不能再显示空 src、破图或错误 alt
sectionActive     当前章节正式接管播放、动画和互动
```

章节可能已经进入视口，但导航或活动观察线仍停留在上一章。此时不能让 `active` 成为挂载图片或 poster 的唯一条件；源挂载由预热状态负责，播放和互动才由活动状态负责。独立预加载器把资源放进 HTTP 缓存，只能证明传输发生，不能替代最终 DOM 消费者设置 `src`。

不能用以下结果证明准备完成：

- 临时创建、随后丢弃的 `<video>`；
- 与页面选择状态无关的 URL 预取；
- 测试脚本手动调用 `load()`；
- 空资源列表；
- 任意旧播放器的 `readyState`；
- 只看网络请求成功但没有核对最终 `currentSrc`。

视频可用状态至少绑定目标 ID、具体节点、`currentSrc` 和可解码事件或 `readyState`。图片可用状态绑定目标 URL、自然尺寸和 `decode()`。预热期间是否暂停、激活时从头还是续播，都写进媒体合同。

图片的 `complete` 在无有效源或加载失败时也可能为真，必须同时满足目标 `src` 或 `currentSrc`、`naturalWidth > 0`、`naturalHeight > 0`，并在需要时等待 `decode()`。视频是否复用同一播放器通过真实节点身份、节点数量、目标 `currentSrc` 和预热状态延续证明，不用 Resource Timing 条目数量代替节点身份；媒体 Range 请求和缓存复用会让该数量在不同浏览器会话中变化。

如果最终播放器尚未建立，不应让另一个无关联元素承担全部预加载工作。可以提前挂载真实消费者并保持不可见，也可以使用能转移响应体或解码结果的明确缓存边界；无论哪种方式，都要证明最终消费者复用了同一资源。

### 5.1 分开导航、视觉过场和预热

三项职责分别建模：

| 职责 | 输入真源 | 可观察结果 | 不能依赖 |
| --- | --- | --- | --- |
| 导航 | 稳定页面注册与用户目标 | URL、页面身份或活动章节改变 | 某个视觉动画是否播放 |
| 视觉过场 | 当前入口与动态效果合同 | 开门、淡出、推入等视觉过程 | 是否需要预热目标资源 |
| 下一目标预热 | 页面顺序、用户已选目标或当前关键就绪状态 | 最终消费者提前挂源并解码 | 可选过场的回调 |

某个入口可以直接导航而不播放过场，但仍复用同一目标解析和预热边界。预热应在页面到达前的稳定事件上触发；减少动态效果、动画取消、动画资源失败或入口明确跳过动画时，不能因此丢失下一页准备。不要建立仅由“动画进行中”状态进入的预加载分支。

## 6. 首帧可见性

poster 或静态锚点和视频属于同一构图。页面首次绘制时先显示 poster，直到目标视频在最终消费者中出现首个可解码画面；随后原子切换可见状态。

不要在页面进入后才创建播放器、再等待请求和解码，同时把舞台留空。预热失败时保留 poster 和主要导航，页面仍然可用。

## 7. 多媒体选择与异步归属

需要多个媒体入口时，所有入口调用同一个选择函数，不直接改 `src`、`currentTime`、可见类或当前 ID。

每次选择使用递增操作编号，并捕获：

```text
operationId
targetId
targetSrc
consumerNode
requestSource
completionPolicy
```

每个异步返回点重新核对这些值。过期操作只能清理自己仍然拥有的旧节点，不能暂停、归零、隐藏或重播已经被新操作接管的消费者。

需要备用播放器时，晋升当前身份、媒体可见性和选择状态应作为一个稳定提交；任一稳定帧最多显示体验合同允许的媒体层数。没有叠加设计目的时，不允许通过半透明交叉显示两个人物来掩盖不对齐。

## 8. 请求仲裁只在确有竞争时建立

若自动媒体、用户选择和导航可能同时请求，先写优先关系和可打断边界：

```text
请求何时算受理：
等待加载期间是否可替换：
播放期间是否可替换：
完成事件：
完成后动作：
失败后用户看到什么：
```

不可打断流程从请求受理时开始锁定，而不是等播放已经开始。锁由统一运行边界维护；按钮的禁用样式只能反映状态，不能成为唯一防线。固定计时只用于识别超时并恢复可用界面，不能冒充真实媒体完成。

普通可替换请求可以合并或保留最后一次有效用户选择。规则来自当前产品合同，不保存某个项目的固定优先级表。

## 9. 循环与转场

循环由真实媒体边界和动作语义决定。运行时只执行已经通过素材验收的方法：

- 直接循环需要首尾画面和动作连续；
- 往返需要反向运动仍自然；
- 匹配切换需要边界相位接近；
- 遮蔽转场需要符合页面视觉语言并保持信息连续；
- 交叉淡化需要叠加时仍只有一个清楚主体。

发现双重轮廓、重影、背景错位或亮度突变时，回到素材边界重新选择，不继续增加混合时长。

## 10. 章节和滚动

只有页面目标需要滚动驱动演出时才建立章节状态。每章先选择唯一架构：

- 普通内容：原生文档流；
- 需要在完整舞台中推进：原生滚动配合 sticky，或固定舞台状态换场；
- 明确的离散局部浏览：局部步进；
- 用户明确接受离散落页：分页吸附。

同一章节不能由全局滚动、局部滚轮、时间线和组件状态同时拥有连续进度。普通页面输入归浏览器；局部组件只在其生效区域和合同范围内消费输入。

需要章内演出时使用：

| 阶段 | 可观察结果 | 章内进度 | 下一内容 |
| --- | --- | --- | --- |
| 进入 | 舞台尚未完整到位 | 0 | 不参与章内演出 |
| 就位 | 关键主体和界面完整可见 | 0 | 尚未释放 |
| 演出 | 状态随唯一进度推进 | 0 到 1 | 不遮断本章 |
| 释放 | 终态稳定 | 1 | 才可进入 |

反向行为、是否重演和已消费距离怎样处理由当前体验合同决定，不写成角色网站的统一默认。

### 10.1 离散滚轮手势

滚轮用于轮播、形态选择或逐项镜头时，输入所有者应按手势会话工作：

```text
连续 wheel 事件
→ 按 deltaMode 归一化并累积有效位移
→ 越过阈值后最多选择一次
→ 吸收同一会话剩余事件与惯性尾段
→ 输入安静达到合同间隔后结束会话
```

阈值控制触发所需意图，空闲间隔区分相邻手势，二者职责不同。不要在每次索引改变后立即解锁，也不要用固定冷却时间吞掉用户已经开始的新手势。到达首项或末项时，本次会话仍由局部轮播消费；会话结束后的新同方向手势才允许页面继续滚动。反向新手势应立即具备选择能力，不受上一方向的惯性锁长期阻塞。

真实验收应按触控板或鼠标实际产生的连续事件节奏发送一组输入，不能在每个 `wheel` 事件之间等待索引变化，否则测试会人为把一个手势拆成多个手势。至少观察一次正常推进、一次惯性尾段、一次快速反向和一次边界释放。

## 11. 分开导航和恢复生命周期

页面加载和返回行为至少分成四类：

| 生命周期 | 进入原因 | 页面与滚动起点 | URL / 历史 | 媒体与互动状态 | 所有者 |
| --- | --- | --- | --- | --- | --- |
| 完整加载或刷新 | 新建文档 | | | | |
| 站内导航 | 当前文档中的用户操作 | | | | |
| 浏览器前进 / 后退 | 历史遍历 | | | | |
| 页面缓存恢复 | 浏览器恢复旧文档 | | | | |

完整加载若必须从开头开始，在浏览器可能恢复旧位置之前由最早启动边界接管，并让应用初始化复用同一结论。不能先在 HTML 阶段归零，又在应用挂载后根据旧 hash、旧章节状态或持久化数据跳回。站内导航和历史遍历可以有不同恢复规则，但每种生命周期只能有一个最终位置所有者。

刷新位置、当前章节、媒体完成状态和“本次会话看过什么”分别建模。它们的生命周期不同，不能由一个全局记忆开关或同一持久化对象代替。

页面改名、合并或移除时，从稳定页面注册真源迁移显示名称、顺序、渲染、导航、URL、历史、媒体预热和所有站内入口。原页面承担的有效互动结果要明确指向新目标或有意取消；迁移完成后移除旧身份、旧恢复分支和旧入口，不留下隐藏兼容页面。

## 12. 目录型内容

如果存在多个形态、作品或媒体项，至少分开：

```text
catalog         完整可访问目录
activeId        当前选择
sessionSubset   仅在体验需要临时抽样时存在
visitedIds      仅在体验需要记录浏览时存在
```

每个入口能访问完整目录还是会话子集由体验合同决定，不让随机抽样自动缩小全部入口。按钮、键盘、滚动和媒体结束事件如果都能选择内容，应复用同一选择边界。

### 12.1 让集合排布随内容变化

多卡片集合把内容状态和排布状态分开：

```text
catalog            正式内容、顺序、媒体和可访问名称
activeId           当前选择
layoutMode         当前视口和动态效果模式
groups             由正式排布函数从 catalog 与 layoutMode 派生
```

排布函数读取条目数量、原始比例、可用宽高和当前模式，输出有序分组；组件只渲染这些分组，不根据条目 ID 或当前位置维护第二份坐标表。新增或移除条目时，目录、卡片、读数、键盘顺序和预热目标沿同一生产链更新，不新增逐项 CSS 选择器。

桌面或宽屏可以使用多行、分列或有意重叠，但要定义水平间距、垂直间距和最大遮挡，并保证标题、说明和操作区域不被其它卡片覆盖。移动端可以把相同分组展平为保持目录顺序的横向滚动吸附轨道；减少动态效果模式改为静态行列或网格。三种模式共享 `activeId`，仅反馈方式和排布不同。

用户要求集合能够继续扩展时，在生产排布函数上验证支持范围内的数量不变量，并在真实浏览器中验证当前正式目录。若未来数量本身是本轮明确交付，浏览器场景必须通过正式内容生产者提供额外条目并让真实组件消费，不能直接追加 DOM、改消费端状态或手写几何结果。

## 13. 声音、可见性和减少动态效果

- 每类媒体单独声明声音职责，不让一个全局布尔值意外修改所有播放器。
- 自动播放媒体只有在合同允许且浏览器策略可满足时启动。
- 离开视口或页面进入后台时，按职责暂停不再需要的媒体；返回时按当前选择恢复。
- `prefers-reduced-motion` 保留完整信息、poster 和导航，移除只为动画存在的等待与滚动距离。
- 触摸设备不能依赖悬停作为唯一入口。

## 14. 真实结果

实现后证明：

- 真实输入经过正式选择边界改变了目标；
- 正式清单产生了真实请求；
- 最终消费者读取并解码了同一目标；
- 直接导航与带视觉过场的入口从同一真源得到目标，跳过或取消过场时预热仍然发生；
- poster 覆盖了首帧前的等待；
- 循环或转场在最终尺寸中没有重影和跳变；
- 触摸、键盘和减少动态效果路径仍能取得同一信息；
- 完整加载、站内导航、历史遍历和页面缓存恢复分别得到合同中的位置与状态；
- 位移动效从合同起点进入，并在相对目标锚点的精确终点稳定下来；
- 离散滚轮输入一次手势最多推进一项，惯性尾段不跳项，边界在下一手势中释放页面滚动；
- 错误时用户看到可用 poster、导航或明确操作，不是空舞台；
- 不存在第二套旧选择、预热、播放或恢复路径。
