# 滚动性能预防与根因诊断

规划或实现角色媒体网站的滚动驱动、固定舞台、Canvas 和多媒体章节时，在编码前使用本文件确定章节类型、滚动架构、生命周期和性能职责；出现滚动掉帧、动画提前、强制吸附感、反向空滚、首次进入章节卡顿或“像是视频拖慢页面”时，再用同一套边界定位根因。目标是阻止互相冲突或所有权不清的架构进入代码，而不是等用户感到卡顿后关闭视频、降低画质或放大滚轮输入。

## 1. 编码前先完成性能职责表

每个滚动章节在实现前填写：

| 章节 | 类型 | 主架构 | 阶段生产者 | 连续进度唯一所有者 | 输入所有者 | 反向策略 |
| --- | --- | --- | --- | --- | --- | --- |
| | 被动 / 分段演出 / 全屏换场 / 局部步进 / 首屏媒体门 | 原生 / sticky / 固定舞台 / 局部输入 / 分页吸附 | | | | |

| 章节 | 离散状态生产者 | 连续进度唯一所有者 | 大面积绘制或合成效果 | 同时活动媒体 | 冷启动预算 | 预热触发与目标身份 | 可用证据与激活行为 | 减少动态效果 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| | IntersectionObserver / 其它边界 | CSS scroll timeline / 单一时间线 | filter / clip / Canvas / 无 | | 首屏必须资源 | 浏览器空闲 / 首屏剧情受理 / 其它；来自正式会话状态 | image decode / video currentSrc + readyState；暂停并在激活时从头播放 | |

然后逐项确认：

- 章节尚未完整到达合同锚点时，章内进度固定为 0；章内演出期间下一页不可见；释放时本章已经稳定在终态。
- 普通章节由浏览器拥有输入；只有首屏媒体门和完整就位的显式局部步进可以在合同范围内有限接管输入。
- Gallery 的完整目录、会话样本和当前形态分别有唯一状态；滚轮只推进样本，按钮、键盘和侧卡按合同访问完整目录，不能因随机抽样缩小全部入口。
- 正向章内进度直接映射滚动位置，不用有时长的追赶动画模拟滚动；离散状态只在阈值跨越时更新。
- 反向返回保持最高已完成状态，并在第一次有效输入时越过视觉不再变化的已消费跑道。
- “当前章节、导航状态、首屏可见性”等离散状态只在边界变化时更新，不在每个滚动帧派发全局状态。
- 每个连续视觉进度只有一个所有者；进入阶段和固定后的章内推进不共用同一进度。
- 每个大面积滤镜、裁切、混合、阴影或 Canvas 循环都有明确作用区域、持续时段和移除后损失；静止内容不保持逐帧计算。
- 同时播放或解码的媒体数量已经确定；离开视口的媒体暂停，下一步媒体按导演计划预载。
- 首次进入需要的控制器和会话选择状态已经在交互前建立；媒体传输可以按合同从首屏剧情受理等真实提前量开始，但不能等到章节到达后才初始化整个模块、抽样状态和资源请求。
- 减少动态效果模式保留完整内容顺序和导航，不保留只为动画服务的固定距离。
- 已经写出真实输入、媒体事件、最终路由和稳定画面的验收条件，不能等实现完成后再决定什么算通过。

任何一项仍写着“之后再看”、存在两个可能所有者，或需要依赖全局滚轮倍率才能让体验成立时，停止编码并先修正合同。

## 2. 出现异常时先分清“慢”和“卡”

这几种问题修法不同：

- **滚动距离过长**：帧率正常，但一个章节需要反复滚轮才能通过。检查章节屏数、信息密度和 `scrub`。
- **动画跟手延迟**：页面持续更新，但滚动位置与视觉进度之间有明显滞后。检查带时长的 `scrub`、重复平滑和多个时间线争用。
- **动画提前**：当前章节尚未完整就位，内部编号、文案或人物状态已经推进。检查进入进度是否误当成章内进度。
- **输入黏滞**：帧率正常，但滚轮或触控板要积累很久才移动，或每次都被拉回某页。检查全局吸附、平滑滚动器、惯性节流和长期输入拦截。
- **反向空滚**：已经完成的章节保持终态，但向上要滚很久才离开。检查已消费演出跑道是否仍保留为无反馈距离。
- **页面交界不符合合同**：内部演出还没结束就露出下一页，或用户禁止吸附却仍要求绝不出现交界。先检查架构取舍和释放边界，不能先调时长。
- **实际掉帧**：滚轮输入后画面停顿或跳跃。检查主线程长任务、同步布局、绘制、光栅、合成和媒体解码。
- **首次进入才卡**：第一次进入章节明显停顿，再次进入正常。检查图片解码、字体、视频首帧、动态 import 和一次性初始化。
- **视频自身掉帧**：页面其它元素仍平滑，但视频画面跳帧。检查真实解码帧、分辨率、码率和同时播放数量。

用户说“滚动不流畅”时，先用真实路径确认属于哪一类；不能把所有问题都称为动画太多或视频太大。

## 3. 固定一条可重复路径

记录：

```text
视口与设备：
减少动态效果：
冷启动 / 热缓存：
起点与终点章节：
具体输入：滚轮次数、触摸距离或导航点击
可见异常：
是否只发生第一次：
```

每次对照必须走同一路径、相同视口和相同媒体状态。不要一次用鼠标滚轮、下一次直接调用滚页函数；也不要以测试代码手写章节状态代替生产者输出。

## 4. 沿一帧的真实链路取证

按以下顺序检查：

```text
滚轮或触摸输入
→ scroll / wheel 监听器
→ 状态更新与组件渲染
→ 样式计算与布局
→ 绘制与光栅
→ 图层合成
→ 图片或视频解码
→ 用户看到的画面
```

在浏览器 Performance 记录中至少观察：

- 主线程长任务和每帧脚本耗时。
- `Recalculate Style`、`Layout`、`Paint` 和光栅峰值。
- 同一滚动是否导致整棵组件树重复渲染。
- 滚动监听器里是否先写样式再读取几何，形成强制同步布局。
- 大面积 `filter`、`backdrop-filter`、`clip-path`、阴影、混合模式和透明固定层是否每帧重绘。
- Canvas 是否在内容静止时仍保持自己的 `requestAnimationFrame` 循环。
- 视频 `currentTime` 是否增长，`getVideoPlaybackQuality()` 的 dropped frames 是否随异常明显增加。
- 首次进入时是否同时发生图片 `decode()`、视频首帧解码、字体加载或大量网络响应。

单看总 FPS 或任务函数名不够。要把性能峰值对应到用户正在经过的章节、正在变化的元素和真实输入。

## 5. 高概率根因和正确边界

### 每次滚动都更新全局状态

典型表现是 `scroll` 监听器在每帧通过 `requestAnimationFrame` 派发 reducer、Context 或顶层 React 状态，即使活动章节和导航状态没有变化也触发组件树工作。

修法：

- 离散状态如“当前章节”“导航是否收紧”“首屏是否可见”使用 `IntersectionObserver` 或只在边界变化时更新。
- 连续视觉进度留在 CSS scroll-driven animation、单一时间线或局部动画层，不经过应用级状态。
- 迁移所有消费者后删除旧 scroll helper、旧 action 和重复恢复逻辑，不保留两套状态真源。

### 进入、章内演出和释放共用一条进度

典型表现是章节只露出一部分时内部动画已经开始，内部最后一段还没出现下一页却已经进入视口。根因不是动画时长，而是把页面进入几何、章内状态和离开释放压进同一个归一化进度。

修法：

- 由统一章节生命周期生产进入、完整就位、章内演出和终态释放四个阶段。
- 完整就位前章内进度硬性为 0；章内演出阶段保持完整舞台；进度到 1 且终态稳定后才释放下一页。
- 所有 Character、Personality 等章节迁移到同一边界后，删除各页面自行计算阈值、恢复终态和补偿延迟的旧逻辑。

### 页面与局部组件争夺输入

典型表现是帧率正常但滚轮不灵敏、方向改变后仍被节流，或 Gallery 切换一次人物后页面长期不动。检查全局吸附、平滑滚动器、页面级 wheel 监听器和局部轮播是否都在消费同一手势。

修法：

- 普通页面输入归浏览器；首屏媒体门只在播放不可打断叙事时接管；Gallery 只在完整就位且向下还有未看形态时有限接管。
- 向上滚动在 Gallery 中始终返回未消费，让浏览器立即移动页面。
- Gallery 的局部滚轮按稳定样本推进；左右按钮、方向键和侧卡可以访问完整目录，但都复用同一目录选择函数，不各自维护媒体、节流和切换状态。
- 活动视频播放由真实可见性控制，不把负向滚动方向当作暂停条件；从下方向上重新进入时恢复当前视频。
- 如果用户没有接受分页手感，移除全局吸附、滚轮累计阈值和强制落页恢复逻辑，而不是降低阈值。

### 章节终态触发依赖指针位置

典型表现是末章卡片已经接近页面底部，但只有鼠标移到视口中间继续滚轮才出现。根因通常是 wheel 监听器挂在局部舞台、上层覆盖层改变命中，或触发条件混入 hover / `event.target`，不是动画时间太晚。

修法：

- “章节已经稳定到终态”由章节生命周期、滚动几何或唯一进度生产者判断，不由鼠标横坐标、hover 元素或局部卡片监听器判断。
- 页面滚轮归浏览器时，不为弹层动画额外长期接管输入；卡片自动出现只消费章节稳定状态。
- 在视口左、中、右使用相同输入复测。如果结果不同，先修监听范围、覆盖层或事件归属，不能单纯提前触发阈值。

### 反向滚动保留已消费跑道

典型表现是页面已经保持终态，向上滚动却要经过很长一段视觉完全不变的距离。正向演出跑道仍存在，但反向又不重播，产生了没有反馈的滚动。

修法：

- 完成章节后把最高进度和终态作为唯一真实状态。
- 折叠不再需要的跑道并补偿视口，或在第一次反向输入时一次性转移到跑道起始边界。
- 只跳过视觉状态完全相同的距离；不能跳过用户尚未读到的内容或首页“欢迎回来”等明确彩蛋。

### 多条滚动时间线承担同一进度

典型表现是多个 ScrollTrigger、手写插值和 CSS transform 同时修改同一元素，或带时长的 `scrub` 在用户停止后仍追赶进度。

修法：

- 一个可见状态只由一个进度边界生产。
- 进入章节和固定后的章内推进分开。
- 纯跟手进度使用无额外追赶时长的映射；只有确实需要惯性表达时才保留延迟，并记录其体验成本。
- 压缩章节高度后重新映射完整就位、章内演出和终态释放，不能只改高度。

### 绘制与合成成本过高

常见来源包括大面积动态模糊、阴影、`backdrop-filter`、复杂 `clip-path`、多层半透明 fixed/sticky 元素、全屏混合模式和过大的 Canvas。

修法：

- 先缩小受影响区域和变化时段，再考虑降低效果强度。
- 静止装饰不维持逐帧动画；能预渲染的静态结果不重复计算。
- `transform` 和 `opacity` 适合合成并不意味着所有元素都应长期 `will-change`；只给实际转场层短期提升。
- 保留用户可见的状态推进，移除没有信息或反馈职责的持续运动。

### 首次资源解码阻塞滚动

如果只有第一次进入章节卡顿，且时间线同时出现图片、字体或媒体首帧解码，问题不是滚轮速度。

修法：

- 分开定义冷启动预算和交互提前量。首屏关键资源先完成，不把整个 Gallery 目录塞进初始下载；但如果开门播放为 Gallery 提供了明确提前量，就在开门请求被受理时预热，而不是机械等到“首屏完全空闲”或 Gallery 已进入视口。
- 预热目标必须来自生产代码的同一媒体清单、会话样本和当前形态：优先本次将显示的人物视频，以及到达时真实可见的当前与相邻海报，不能由测试或独立 helper 复制一份静态列表。
- 对即将进入的图片调用真实 `decode()`；视频设置目标 `src`、调用 `load()` 并保持暂停，避免所有形态同时播放或争抢解码器。激活时重新核对目标身份、归零并按合同播放。
- 可用状态由真实资源证明：图片有自然尺寸且解码完成；视频的 `currentSrc` 对应目标，并达到目标的 `loadeddata/canplay` 或 `readyState >= 2`。空预热列表是 idle，不是 ready；旧请求的 ready 事件不能满足新形态。
- 预热失败不阻塞导航；页面仍以同一形态的海报或锚点可用，不能暴露空舞台。按“触发 → 请求发出 → 解码可用 → 到达章节 → 目标可见”记录时序；只证明最终播放成功无法排除到达后的空白等待。

### 视频解码确实是主因

只有同时满足以下证据时，才优先调整视频：

- dropped frames 与用户看到的卡顿同步增加。
- 主线程脚本、布局和绘制没有更大的峰值。
- 用同尺寸静态海报替代视频后，同一路径明显改善。
- 问题与分辨率、码率、编码格式、透明视频合成或同时解码数量存在稳定对应关系。

此时再调整分辨率、码率、编码、预加载和同时播放策略。不能因为文件较大就直接断言它是滚动卡顿的主要原因。

## 6. 用单变量对照收口根因

每次只改变一个边界：

1. 保持视频不变，停止每帧全局状态更新。
2. 保持逻辑不变，临时关闭一个高成本滤镜或 Canvas 循环。
3. 保持页面结构不变，用海报替代正在播放的视频。
4. 比较冷启动与热缓存，确认是否为首次解码。
5. 在桌面、移动端和减少动态效果模式复测同一路径。

对照只用于定位，最终修复必须落回正式架构。不能把临时关闭类、测试分支或两套实现留在生产代码里。

## 7. 修复顺序

按主要成本从上游收口：

1. 先确认章节类型和滚动架构，解决互相制约的体验要求；不在错误架构里继续调阈值。
2. 建立进入、完整就位、章内演出、终态释放和反向返回的唯一生命周期，迁移所有章节消费者并移除旧阈值与恢复逻辑。
3. 删除重复生产者和无变化的状态更新，把连续视觉进度移出应用级状态，交给局部时间线或浏览器原生滚动动画。
4. 明确输入所有权，移除未授权的全局吸附、滚轮累计和长期输入拦截；保留首屏媒体门与 Gallery 的合同级例外。
5. 减少大面积逐帧绘制、同步布局和无职责循环。
6. 按最接近的真实使用期限预热下一章节：冷启动不争抢首屏；合同允许时从首屏剧情受理开始准备 Gallery 的会话选中项和可见海报。
7. 有解码证据时再处理视频规格和播放数量。
8. 重新走页面顺序、开门剧情、Gallery、目录、返回首屏和减少动态效果链路，确认性能修复没有改变功能终态。

不要通过增加滚轮倍率、长期 `preventDefault()`、强制翻页或删掉有信息价值的状态来制造“更快”的感觉。

## 8. 通过标准

交付前必须能够说明：

- 每章的类型、主架构、阶段生产者、输入所有者和反向策略是否唯一，是否与用户接受的手感一致。
- 完整就位前章内进度是否为 0，章内演出期间下一页是否不可见，释放时是否已到终态。
- 反向第一次有效输入是否立即有反馈，是否已经移除视觉不变的已消费距离。
- Gallery 是否只用滚轮消费有限样本、向上立即释放，完整目录是否仍能由按钮、方向键和侧卡访问，所有入口是否共用同一目录选择与媒体边界。
- 章节终态和自动弹层是否与鼠标横向位置无关，是否已移除局部 hover 或 `event.target` 对全章触发的控制。
- 编码前的性能职责表是否完整，实际实现是否遵守了同一组所有者和资源时序。
- Gallery 预热请求是否在用户到达前由生产代码发出，目标是否与当前会话形态一致，激活前是否保持暂停且激活时从合同起点播放。
- 资源 ready 是否由目标请求、目标节点和真实解码状态证明；是否不存在空清单、旧事件或任意 active video 冒充当前目标已就绪。
- 冷缓存下从预热触发到 Gallery 可见的时间线是否没有空白窗口；失败时是否立即显示同一形态海报或锚点。
- 主要成本位于哪一层，证据是什么。
- 哪个对照排除了视频、动画、状态或资源预热中的其它候选。
- 修改了哪个唯一生产者或消费边界，旧路径是否已经移除。
- 冷启动和热缓存下的同一路径是否都复测。
- 桌面、移动端和减少动态效果模式是否仍可用。
- 开门动画、章节顺序、导航状态和最终滚动落点是否仍与体验合同一致。

报告先写用户能观察到的结论，再写根因证据、修复和未验证项。只写“优化了动画”“减少了重绘”或“感觉更流畅”不算通过。
