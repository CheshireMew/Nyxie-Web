# 滚动性能与所有权

用于包含 sticky、固定舞台、滚动驱动动画、Canvas 或多媒体章节的网站。先建立滚动与渲染职责，再编码；出现问题时沿同一条真实链路定位根因。

## 1. 编码前合同

| 章节 | 页面目的 | 主架构 | 输入所有者 | 连续进度所有者 | 离散状态生产者 | 活动媒体 | 预热时机 | 减少动态效果 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| | | | | | | | | |

可选架构：

- 原生文档流：阅读和普通内容；
- 原生滚动加 sticky：完整舞台中的连续演出；
- 固定舞台状态换场：需要过渡中保持单一完整舞台；
- 局部步进：用户明确逐项浏览的离散交互；
- 分页吸附：用户明确接受离散落页手感。

一章只选一个主要架构。普通页面输入归浏览器；局部输入只有在生效区域、方向和未完成状态均符合合同时才消费。

需要章内演出时再填写：

| 阶段 | 页面几何 | 章内进度 | 下一内容 | 反向行为 |
| --- | --- | --- | --- | --- |
| 进入 | 舞台尚未完整到位 | 0 | 不参与演出 | |
| 就位 | 关键主体完整可见 | 0 | 尚未释放 | |
| 演出 | 舞台保持合同构图 | 0 到 1 | 不破坏本章 | |
| 释放 | 终态稳定 | 1 | 可以进入 | |

反向是否保持终态、重演或折叠已消费距离由当前页面目标决定，不保存统一默认。

## 2. 性能职责

逐项确认：

- 连续视觉进度只有一个生产者；
- 离散状态只在边界变化时更新，不在每一帧重复派发相同全局状态；
- 输入、状态、布局、绘制、合成和媒体解码的职责清楚；
- 大面积滤镜、混合、裁切、阴影和 Canvas 循环都有明确作用区域与持续时间；
- 静止内容不保持无意义逐帧工作；
- 同时播放和解码的媒体数量有上限；
- 即将使用的媒体按正式选择状态预热；
- 减少动态效果模式保留内容顺序，移除只为动画存在的距离。

存在两个进度所有者、两个输入所有者或依赖全局滚轮倍率才能成立时，不进入实现。

## 3. 区分问题类型

| 现象 | 优先检查 |
| --- | --- |
| 滚动距离长但帧率正常 | 章节高度、信息密度、无变化跑道 |
| 视觉进度追不上滚动 | 带时长的追赶、重复平滑、多个时间线 |
| 章节尚未就位就开始演出 | 进入进度和章内进度混用 |
| 输入黏滞或方向改变后仍被拦截 | 吸附、全局 wheel、惯性阈值、局部输入所有权 |
| 第一次进入才卡 | 图片、字体、视频首帧、动态模块初始化 |
| 页面元素都卡 | 主线程、布局、绘制、光栅、合成 |
| 只有视频画面掉帧 | 解码、分辨率、码率、透明合成、同时播放数量 |
| 页面到达后先空白 | 预热触发、最终消费者和 poster |
| 下一章节已露出但仍是破图或替代文字 | 资源是否只进了缓存、最终节点是否把挂载条件错误绑定到 active |

先判断是哪一种，不把所有问题归因于视频文件大或动画太多。

## 4. 固定真实复现路径

```text
视口与设备：
冷缓存 / 热缓存：
减少动态效果：
起点与终点：
真实输入：
媒体状态：
可见异常：
是否只发生第一次：
```

每次对照使用相同路径、视口和媒体状态。不能一次用真实滚轮，下一次直接调用滚页函数；不能手写进度或 `ready` 状态代替生产链路。

## 5. 沿一帧取证

```text
真实输入
→ 输入监听和浏览器滚动
→ 状态更新与组件渲染
→ 样式计算与布局
→ 绘制与光栅
→ 图层合成
→ 图片或媒体解码
→ 用户看到的画面
```

浏览器 Performance 中观察：

- 主线程长任务和脚本耗时；
- 组件是否因连续输入重复渲染；
- 强制同步布局；
- 大面积滤镜、混合和透明固定层；
- 内容静止时仍运行的 Canvas 或 `requestAnimationFrame`；
- dropped frames 是否与可见卡顿同步；
- 首次进入时是否集中发生资源请求和解码。

总 FPS 和单个函数名不足以说明根因，要对应到真实章节、真实输入和正在变化的元素。

## 6. 从上游修复

按证据收口：

1. 先修正章节目标和滚动架构；
2. 确定唯一输入与进度所有者；
3. 分开进入、章内演出和释放；
4. 把连续视觉值留在局部时间线或 CSS，不经过全局应用状态；
5. 缩小高成本绘制范围和持续时间；
6. 从真实提前量预热即将使用的最终媒体；
7. 只有解码证据成立时才调整视频规格；
8. 一次性迁移消费者，移除旧监听、旧 action、旧 helper 和旧恢复逻辑。

不能用加大滚轮倍率、长期 `preventDefault()`、强制翻页或删掉有信息价值的状态制造“更快”的假象。

## 7. 资源预热专项

首次进入卡顿或先空白时，记录：

```text
预热触发
→ 正式选择状态
→ 目标 URL
→ 请求开始
→ 最终消费者或可晋升备用消费者可解码
→ 内容首次可能进入视口
→ 页面到达
→ 目标可见
```

图片通过目标 URL、自然尺寸和 `decode()` 证明；视频通过目标 ID、节点、`currentSrc` 和目标媒体事件证明。临时元素、旧事件、测试预取和空列表不能冒充准备完成。

滚动章节分别记录预热触发线、内容首次进入视口的位置和活动章节观察线。最终媒体源应在预热触发后挂载，不能等活动线切换后才设置；活动状态只负责开始播放、入场动画和互动。网络面板已经下载资源但最终节点仍为空源，属于消费者断链，不是预加载成功。

预热不能与首屏关键资源无上限竞争，也不能拖到页面已经进入后才开始。准确触发点来自页面结构和用户行为提供的真实提前量。

## 8. 通过标准

- 每章的架构、输入所有者和进度所有者唯一；
- 章内演出只在合同构图完整后推进；
- 正向每段距离都有信息、构图或反馈变化；
- 反向行为与当前合同一致，没有无职责距离；
- 冷缓存和热缓存都走同一正式链路；
- 资源在需要前由生产代码请求，最终消费者复用同一目标；
- 能用对照证据说明主要成本所在层；
- 修复没有改变页面目标、媒体职责、互动结果和最终导航；
- 被替代的旧实现已经不在有效代码路径中。
