# 角色视觉与视频素材工作流

用于从角色设定图建立网站锚点，并把首尾帧不可控的生成视频处理成可用网页素材。

## 1. 先提取角色视觉真源

从设定图只提取会影响一致性的内容：

- 脸型、眼睛、发型、耳朵、肤色和身体比例。
- 服装轮廓、层次、材质和不可缺失的配饰。
- 角色的标志物、主色和辅助色。
- 性格动作：警惕、顽皮、优雅、笨拙、亲近或疏离。
- 原型带来的行为逻辑，例如引路、恶作剧、隐藏、观察或守护。

把特征分成三层：

| 层级 | 内容 | 生成时是否允许改变 |
| --- | --- | --- |
| 身份锚点 | 脸、发型、服装、核心配饰 | 不允许 |
| 风格锚点 | 材质、渲染方式、光线、色彩关系 | 小范围调整 |
| 场景变量 | 姿势、镜头、道具、局部环境 | 可以按片段改变 |

不要把“Q 版”“二次元”“3D”等单一标签当作身份锚点。用户允许改变比例或渲染风格时，保留身份特征即可。

生成图不能反过来改写原设。锚点图中新出现的道具、门、道路、月亮、机械结构或其它装饰，默认仍是场景变量。只有用户明确确认后，才能升级为角色标志物或在其它章节反复使用。每次扩展素材前检查：这个元素来自原设、用户决定，还是模型偶然生成？

### 逐图证据卡

用户给出多张人物图并要求“每张分别写提示词”时，先按实际文件逐张观察，不从上一张的主题续写下一张，也不根据列表位置猜身份。每张至少记录：

```text
图片 ID 或原文件名：
可见的脸、发型、耳朵和身体比例：
必须保持的服装、材质和配饰：
当前姿势、重心、视线和手势：
画面实际存在的翅膀、尾巴、特效或物品：
人物性格由哪些可见细节支持：
没有出现、不得自行补写的元素：
适合继续发展的动作瞬间：
```

形态名称只有三个合法来源：用户明确命名、原文件名、图片中能直接确认的既有设定。没有来源时使用“第 2 张”“黑色礼服形态”等中性描述，不编造“梦境导航员”这类身份。道具也必须逐项写明来源；不能因为奇幻角色常见钥匙、月亮或星星，就把它们塞进每张图。

## 2. 写视觉方向，不堆风格词

视觉方向需要回答：

1. 角色是什么气质。
2. 世界是什么材质和光线。
3. 首页是明亮、神秘、危险还是轻盈。
4. 角色与网页文案各占什么区域。
5. 第一视觉焦点是什么，第二焦点是什么。

色彩必须写角色分工，而不是只列颜色：

```text
主背景：高明度冷白与浅天蓝，约占 70%
角色主体：黑色与银白，约占 25%
强调色：红色和少量金色，约占 5%
避免：大面积紫色、品红色环境光、纯黑背景
```

生成模型经常用紫色表达“奇幻、暗黑、神秘”。不想要紫色时，要同时指定主背景、强调色比例和排除项。只写“不要紫色”通常不够。

要求立体感时，不要只写“3D”。明确：

- 角色是高质量 3D 动画电影渲染、半写实游戏 CG，还是保留二维脸部设计的 2.5D。
- 布料、金属、头发、皮肤分别是什么材质。
- 使用轮廓光、环境反射、体积光和前中后景景深。
- 镜头焦段、机位高度和透视关系。

### 把“有设计感和张力”写成可执行构图

动态人物默认从以下关系中选择两到四项，不全部堆进一张图：

- 身体重心落在一条腿或一个明确接触点上，肩胯方向有轻微反差。
- 头、视线、躯干和手势不全部平行，形成清楚的动作方向。
- 使用前后遮挡、近大远小或对角线，而不是正面站直、双臂平均展开。
- 手必须有具体意图：扶住、回望、准备迈步、拨开、收回或触碰真实物体。
- 给动作方向留出负空间，避免人物被均匀塞在画面中央。

“自然、优雅、有张力”不能替代重心、接触点、视线和手势说明。用于角色设定的标准正身图可以稳定中立；用于性格和动作表现的画面必须有明确瞬间。

动作方向从该图可见性格和服装结构出发，不从通用姿势库平均分配。恶魔形态可以通过控制火焰、收拢或展开翅膀、尾巴反向平衡表达压迫感；花庭形态可以通过转身牵动裙摆、接住花瓣或回望表达轻盈。不能把所有形态都写成“正面站立、平视或轻微仰视、一只手抬起”。同一批图还要主动拉开剪影：正面与背身、下沉与跃起、向前逼近与向后回望、收束与展开之间至少形成可辨差异。

## 3. 设计首页锚点图

锚点图至少满足：

- 角色脸和关键动作位于桌面与移动端都能保住的区域。
- 固定导航不会遮住头部、耳朵或标志物。
- 文案区域有足够低细节背景和明暗对比。
- 配饰服从角色层级，不抢脸。宝石、徽章或武器过亮时，限制尺寸、发光强度和饱和度。
- 环境体现角色原型。例如“童话型恶作剧角色”应通过不合理尺度、悬浮路径、错误的门、视觉谜题或观察感体现，不只添加月亮和星星。
- 画面本身可作为视频加载失败时的完整首页。

锚点负责首页构图和风格 DNA，不自动成为全部页面的场景模板。下方章节可以更换明暗、空间和构图，只要继续使用相同的色彩比例、材质关系、字体气质、线条和运动节奏。

### 锚点图提示词结构

保持精确，按以下顺序写：

```text
[角色身份锚点]。
[比例、姿势、表情和角色位置]。
[世界设定和能突出角色性格的场景元素]。
[渲染方式、材质、镜头和景深]。
[主色、辅助色、强调色比例和光线]。
[网页构图：比例、文字安全区、导航安全区]。
[必须保留和必须避免的内容]。
```

不要一次加入互相竞争的十几种风格、太多道具或多个主要光源。每次迭代只修正会改变网站表现的问题。

## 4. 用锚点生成视频

当生成器只能使用首帧时：

1. 所有视频使用同一张锚点或经过严格对齐的变体。
2. 每段视频只安排一个主要动作和一个次要自然动作。
3. 明确相机是固定、轻推、跟随还是后退；默认固定或轻微推进最适合网页叠加文字。
4. 不要求结尾精确回到首帧。
5. 把动作高潮安排在时长中前段，给尾部留出裁剪空间。
6. 写明角色全程保持完整服装、年龄表达、身体结构和非性化表演。
7. 消失、变身或召唤替代形象时，明确替代形象的身份特征、画风、材质和出现顺序。

“同一张锚点”只约束同一形态、同一主舞台里需要连续切换的片段。多形态 GALLERY 中，恶魔、花庭、星海等每个形态使用自己的逐图证据和动画首帧；不能为了统一而让八个形态共用一张外观锚点。每个形态的 WebM 与 poster 必须成对，媒体清单负责映射。

生成器能力表是本轮素材计划的唯一依据。用户已经确认只能选择某些时长、只能提供首帧或不能输出透明背景时，后续所有提示词和清单都沿用该限制；不凭经验写入不存在的选项。

### 视频提示词结构

```text
保持首帧角色的脸、发型、服装、配饰、身体比例、渲染风格和环境不变。
动作：角色先……，随后……，主要动作在前半段清楚完成。
细节：只允许头发、衣摆、耳朵、尾巴、道具和环境产生符合动作的次级运动。
镜头：固定机位或非常轻微的……，不切镜，不突然改变焦段。
结束：动作完成后保持自然状态，不要求精确回到首帧。
安全：全程完整服装、非性化姿势，无裸露、无身体变形、无额外人物。
禁止：身份漂移、风格突变、错误道具、镜头乱转、文字和水印。
```

### 完整提示词包标准

交给用户在生成器网页中使用的每条提示词必须独立成立，至少包含：

1. 使用哪张上传图作为唯一角色或构图参考。
2. 足以锁定身份的脸、发型、耳朵、服装、配饰和比例；用户指定的固定身份段按原文保留。
3. 本张图或本段视频唯一的构图、动作、表情和镜头任务。
4. 背景或绿幕要求、比例、留白和元素分离方式。
5. 渲染、材质和光线中真正影响结果的要求。
6. 针对本任务的排除项。
7. 这份素材供哪个页面、首帧或视频使用。

不能写“沿用前面的身份段”“使用某母版”“参考 anchor 图”后让用户自行拼接。可以写“使用我上传的原始角色设定图作为唯一角色参考”，但仍需点名容易漂移的身份特征。提示词追求完整和精准，不用十几种风格词重复表达同一件事。

用户给出的固定渲染段、服装禁区或角色转换要求，必须完整出现在每一条需要它的提示词中，不改成“同上”。但完整不等于堆词：逐图观察没有支持的道具、形态标题、情绪和特效应删除。用户要求“给我完整提示词”时，默认直接给可复制正文，不再让用户自行拼接身份段、负面段或比例说明。

一次需要多张图或多段视频时，按素材 ID 一次性交付全部完整提示词，并在每条前写清：输入图、输出职责、静态用途或后续视频用途。

## 5. 选择静态图、视频首帧或绿幕母图

### 素材入场检查

生成或拆分素材前，逐项回答：

1. 它服务角色身份、内容证据、构图层级还是交互状态？
2. 它与旁边的角色、项目、文案或操作有什么可见关系？
3. 移除它后，用户会少理解什么、少完成什么，或哪个构图关系会失效？
4. 它是否只是为了填空、呼应一个偶然生成的道具，或让画面“更丰富”？

前三项没有明确答案，或第四项成立时，不生成、不拆分、不进入页面。留白可以承担聚焦、节奏和层级职责，不需要用首饰、星月、粒子或悬浮物填满。已经随绿幕母图生成出来的物件也不自动获得使用资格；批次里存在不等于页面需要。

先按用途选择生产方式：

| 目标 | 首选方式 | 原因 |
| --- | --- | --- |
| 正身设定、服装和配饰细节 | 宽幅绿幕母图，人物与细节分区生成 | 便于一次抠图、拆件和保持同批次风格 |
| 性格、转身、靠近、回头等动作瞬间 | 先生成视频首帧，再生成动作视频 | 视频模型更适合连续重心和自然过渡 |
| 动作静态图 | 从合格视频中选帧 | 避免静态模型生成僵硬的“展示姿势” |
| 与项目内容有真实关系的截图、界面或角色元素 | 优先使用真实项目截图、过程图或源文件；需要角色元素时单独生成 | 先证明项目，再用角色建立叙事关系 |

### 宽幅绿幕母图要求

- 背景使用单一、均匀、无渐变的纯绿色；角色和物件本身不能使用接近背景的绿色。
- 人物、边框和已通过素材入场检查的界面元素之间保留足够间距，不重叠、不用投影连接。
- 每个元素完整露出，不贴边；需要拆分的部件分别占据独立区域。
- 背景上不生成文字、透明棋盘格、环境地面、绿色反光和复杂阴影。
- 人物仍需有自身材质、体积和接触逻辑，但不把接触阴影烘焙到绿幕背景。

生成器不能输出透明背景时，明确采用“绿幕生成 → 抠图 → 边缘和溢色检查 → 网页合成”，不要让模型伪造棋盘格透明背景。

### 动作视频工作流

1. 先确定页面要表现的性格瞬间和视频结束后的用途。
2. 生成一张动作已经蓄势、但尚未完成的首帧；写清重心、视线、手势和镜头。
3. 把这张绿幕图作为视频首帧，使用生成器真实支持的时长。
4. 视频提示词只安排一个主要动作和少量次级运动，不切镜。
5. 验收成片后，视频用于动态章节；需要静态图时从自然帧选取。

### 视频首帧的运动安全区

首帧不是把人物尽量塞满画面，而是为接下来 6 秒左右的动作保留运动包络：

- 比例来自最终网页舞台，不默认使用 9:16。竖向全身人物通常选择比 9:16 更宽的 4:5、3:4 或与实际容器一致的比例，再由网页决定裁切。
- 先包住完整剪影：头发最高点、角、耳朵、两侧翅膀、伸展手指、裙摆、尾巴和手持特效都必须在画内；主角不能因为装饰框而缩小。
- 静止状态下四周通常保留约 8%–12% 安全区；动作将朝某一方向展开时，该方向额外留空间。数值只用于首轮构图，最终以真实网页容器和动作幅度为准。
- 首帧停在“动作将要发生”的瞬间：重心已经偏移，视线和手势已经指向目标，但翅膀、火焰、裙摆或身体尚未到最大展开。这样视频模型有清楚的运动方向，而不是从僵直站姿凭空造动作。
- 人物周围只保留一个主要特效和至多两组弱次级元素；每件物品都要服务形态特征或动作路径。碎晶、链条、羽毛、星点等不能均匀撒满全画面。
- 绿幕必须连续、纯净、无地面渐变和投影；人物自己的接触关系可以通过鞋底姿态和衣料受力表达，不把大块阴影烘焙在绿底上。

通过标准不是“人物完整”这一项，而是预演主要动作后，头、手、裙摆、翅膀、尾巴和特效仍不会撞边，同时人物在网站主舞台上保持足够视觉占比。

## 6. 常用视频职责

| 职责 | 推荐生成时长 | 动作设计 | 网页用途 |
| --- | --- | --- | --- |
| 主待机 | 10 秒 | 呼吸、眨眼、观察 | 自动表演起点 |
| 次待机 | 10 秒 | 操作标志物、转移视线 | 打破机械循环 |
| 直接回应 | 6 秒 | 朝镜头递、抛、指或招手 | 空白或角色点击彩蛋 |
| 幻隐动作 | 6 秒 | 角色按自身设定消失或变化 | 随机表演 |
| 章节动作 | 6 秒 | 门、通道、跃迁或场景开启 | 滚动剧情门控 |
| 靠近回应 | 6 秒 | 角色向镜头靠近、邀请或挑衅 | 返回首屏或对话入口 |

时长由生成器能力决定，网页使用时长由真实内容决定。六秒成片可以裁成三秒，十秒成片也可能完整保留。

## 7. 逐段验片

不要只看封面。至少检查开头、25%、50%、75% 和尾部，并播放动作高潮前后。

记录：

```text
文件：
生成时长：
可用起点：
可用终点：
主要动作发生时间：
身份漂移：无 / 轻微 / 不可用
结构或裸露问题：
镜头问题：
适合的互动职责：
是否需要重新生成：
```

处理顺序：

1. 身份或安全问题影响主要片段：重新生成。
2. 只有尾部失控：裁掉尾部。
3. 动作开始太慢：裁掉静止开头，但保留足够切入时间。
4. 镜头轻微漂移：评估网页裁切后是否仍可用。
5. 动作与计划不同但质量高：按真实动作重新分配职责，不强行使用原文件名。

绿幕图片和视频额外检查：背景是否均匀、头发和荷叶边等细边是否可抠、绿色溢色是否污染皮肤和服装、元素之间是否真的可分离。验收结果分为正式使用、拆分利用、重新生成和不使用；整张不合格不等于其中所有部件都无用，但每个准备拆出的局部仍要单独通过素材入场检查。

## 8. 统一网页媒体

先用 `ffprobe` 获取真实信息：

```powershell
ffprobe -v error -show_entries format=duration:stream=codec_name,width,height,r_frame_rate -of json input.mp4
```

常用网页目标：

- 1280×720 或与首页一致的 16:9 尺寸。
- H.264、`yuv420p`、24 或 30fps。
- 无需声音时移除音轨。
- 使用 `faststart`，让 MP4 元数据位于文件开头。
- 同一组视频保持一致尺寸、帧率和像素格式。

示例命令，按真实裁剪点替换参数：

```powershell
ffmpeg -ss 0.20 -to 5.70 -i input.mp4 -vf "scale=1280:720:force_original_aspect_ratio=increase,crop=1280:720" -r 24 -c:v libx264 -crf 22 -preset medium -pix_fmt yuv420p -movflags +faststart -an output.mp4
```

编码后必须再次播放输出文件。不能因为命令成功就认为画面、结尾和浏览器兼容已经通过。

### 从用户视频提取独立 BGM

用户指定“从某个视频的 A 秒到 B 秒提取并循环”时，原视频和精确起止时间是音频生产真源：

1. 用 `ffprobe` 确认原文件存在音轨、真实时长和编码；结束点超出源文件时先报告，不自行缩短。
2. 用精确时间裁取音轨并输出网页支持的音频格式。需要准确边界时重编码音频，不依赖可能偏到关键帧的容器拷贝。
3. 再次读取输出的真实时长，允许的误差在项目合同中写明；从头、区间中点和片尾实际试听，确认没有带入区间外内容、突发静音或损坏。
4. 在媒体清单记录原视频、起止时间、输出文件、MIME、真实时长、音量、默认开关和循环职责。人物视频是否静音是另一列，不能因为存在 BGM 就保留人物视频音轨或让一个声音按钮同时改两类媒体。
5. 网页真实播放接近片尾后观察循环回到起点。下载器接管音频直链时，网页改用 `fetch + Blob URL` 消费同一输出文件；不重新生成另一份内容，也不要求用户修改下载器全局设置。

示例命令，按用户给出的真实区间替换：

```powershell
ffmpeg -ss 00:00:00.000 -to 00:00:59.270 -i input.mp4 `
  -vn -c:a aac -b:a 192k -movflags +faststart output.m4a
```

循环是否听起来无接缝由源区间决定。用户只要求准确截取时不擅自加淡入淡出；用户要求无缝循环时再检查波形边界并取得允许的处理范围。

### 透明人物视频

需要网页透明背景时：

1. 在视频软件中完成去绿、边缘收缩、去溢色和必要遮罩。
2. 不需要声音时移除音轨。
3. 保留一份高质量带 Alpha 母版，再转换网页版本。
4. 用支持 Alpha 的解码器实际读取 Alpha 平面，检查透明与不透明像素分布；不能只看扩展名或 `alpha_mode` 标签。
5. 在目标浏览器和真实章节背景上播放，确认没有黑底、绿边或错误混合。
6. 为不支持该透明编码的浏览器准备静态海报或其它明确降级。

透明静态图也要读取真实 Alpha 通道。图片查看器显示绿色、黑色或棋盘格都不能单独证明文件有没有透明度。

#### 清理透明 MOV 转 WebM 的低 Alpha 方框

透明母版即使带有效 Alpha，也可能让整幅画布残留 `alpha=1..8` 的微弱像素；在浅色网页上会形成肉眼可见的矩形。正确边界是母版到第一次有损编码之间：

1. 保留原始 ProRes 4444 或其它高质量 Alpha 母版，不从已有 WebM 再转码。
2. 使用本 Skill 的 `scripts/clean_alpha_video.py`，先把 `0 < alpha <= 8` 清零，再编码 VP9 WebM。
3. 同步从清理后的 WebM 重做透明 WebP 海报；不能只换视频，留下带方框的旧 poster。
4. 脚本用正确解码器读取母版和 WebM 的真实 Alpha，比较全画面、外圈和四角覆盖，并确认帧数不变。
5. 在网站实际浅色或纹理背景上播放全部形态；统计下降只说明清理发生过，不能代替肉眼确认。

示例：

```powershell
python skills\immersive-ip-website\scripts\clean_alpha_video.py `
  "D:\media\character-alpha.mov" `
  "D:\media\character-clean.webm" `
  --poster "D:\media\character-clean-poster.webp" `
  --report "D:\media\character-clean-alpha.json" `
  --ffmpeg "D:\Tools\ffmpeg\bin\ffmpeg.exe"
```

阈值 8 是首轮值，不是角色无关的常数。如果方框仍可见，可以从原母版尝试 16；每次都必须重新检查发丝、薄纱、翅膀边缘、尾巴、细链和粒子。VP9 有损 Alpha 可能在真实边缘附近重新产生少量 `1..8` 像素，因此输出的 `low_alpha_pct` 不要求机械归零；通过标准是四周不再形成连续画布、主体细节保留、真实页面没有矩形边界。

低 Alpha 方框和黑色半透明边缘不是同一个问题。前者先清 Alpha；后者只有在人物边缘实际出现黑底污染时，才从母版抽帧做颜色修复。不要默认同时提亮黑发、黑衣或深色翅膀。
