# Immersive IP Website 正向行为样本

只在维护本 Skill 的 description、默认流程、边界或 reference 路由时读取。这里用少量正常成功请求检查 Skill 会不会在问题发生前建立正确合同；网站实现阶段不读取，也不把历史纠偏继续积累成故障目录。

## 核心成功请求一：增量修复现有角色站

用户请求：

> 保留现在已经满意的页面效果，统一修复这些问题：正常加载只能看到原来的加载进度界面再进入首页，故障降级页不能先闪出来；首屏开门动作每次完整加载只播一次，返回顶部不能重播，而且等待切换、播放和滚页时都不能被点击或导航打断；开门受理后就预热本次 Gallery 会显示的人物和可见海报，到达时不能空白；Gallery 每次滚轮随机展示 3 个形态，但左右按钮、方向键和侧卡仍能访问全部 8 个，向上回来时人物视频不能静止，切换过程不能两个人物重叠，真实 ended 后按同一选择边界切下一人物；普通人物视频 2×、Gallery 1.5×、开门 2×，Personality 暂时不动，人物视频全部静音，声音按钮只控制默认关闭并循环的 BGM；末章弹层外不能点到底层 GitHub，卡片外也没有链接；刷新回首页顶部；导航和首屏没有间隙；鼠标外圈保留弹簧但不要离红点太远；IDM 不再接管 BGM。旧 Git 版本很久以前就过时了，请按当前页面、截图和我逐项确认的行为核对。请查根因后修复并真实验收。

预期第一步：

1. 判断这是授权增量修复，先把旧 Git 标为不可作为重构前体验证据，改用当前运行结果、用户截图和逐项确认冻结桌面、移动、HTML 首帧、加载态、Gallery 构图和已接受的 Personality 行为。
2. 把要求还原为六份合同：正常启动阶段、媒体职责与完成策略、完整页面加载语义、Gallery 完整目录与随机样本、资源预热时间线、交互几何与命中。
3. 从当前媒体清单、导演、章节状态、DOM 层级、CSS 布局和真实浏览器复现每个问题，沿“生产者 → 状态或传输边界 → 消费者 → 可见结果”确定根因。
4. 找到唯一媒体清单、独占锁、页面会话状态、目录选择、导航高度、弹层遮罩和光标定位边界，一次性迁移所有入口；不从提前阈值、重复阻止冒泡或额外计时开始。

允许的默认动作：读取本地代码、媒体、截图和工作区状态；用户已经授权修复时可以修改站点，运行类型检查和真实浏览器验收。删除素材、安装依赖、打包、发布、推送和修改 IDM 全局设置仍需用户明确要求。

预期输出开头：先说明用户可见结果和真实根因，再说明唯一边界怎样修改；不能只写“调快了动画”“加了 preventDefault”或“测试通过”。

停止条件：所有授权行为都由生产代码生成，真实刷新、滚轮、点击、键盘、媒体事件和网络请求经过正式消费者，用户在桌面、移动和减少动态效果路径能观察到合同结果；未点名区域没有退化。

可观察通过：

- 整页刷新回首页顶部并重新允许一次开门；同一文档返回顶部后向下原生滚动，不重播开门。
- 独占请求从受理到导航稳定都不被动作、导航、键盘或重复滚轮打断。
- 正常启动从合同中的 HTML 加载壳进入加载进度再进入首页，不闪现故障降级页；入口真实失败时仍能进入降级结果。
- 开门请求受理后由生产代码请求本次会话选中的 Gallery 人物和可见海报，目标在到达前可用；预热视频保持暂停，激活时从头播放，冷缓存到达没有空白。
- Gallery 随机样本保持 3 个，但按钮、键盘和侧卡实际加载样本外形态；向上重新进入时活动视频播放。切换按目标 id、节点和 `currentSrc` 等待，稳定画面最多一个人物可见；真实 `ended` 通过同一目录选择边界切换下一人物。
- 每类媒体使用合同中的独立速度，开门为当前确认的 2×，人物视频始终静音；BGM 默认关闭、点击后循环，播放器只暴露内部 Blob URL。媒体清单、实现、测试和本样本里没有遗留历史速度或旧 loop 断言。
- 弹层外点击命中遮罩，卡片外和底层作品不打开新页面。
- 导航下沿与首屏媒体上沿贴合；滚动条按要求隐藏但原生滚动可用。
- 光标快速移动时外圈有弹簧滞后但不超过上限，停止后回到红点，hover 放大不改变圆心。

## 核心成功请求二：在编码前主动预防同类问题

用户请求：

> 我准备做一个带加载进度、首屏剧情、八种人物形态、人物视频、独立 BGM、末章链接弹层和自定义光标的角色网站。首屏剧情每次刷新只播一次，并在播放期间预热 Gallery；Gallery 滚轮随机看三种，但按钮要能看全部，人物结束后自动切下一种。怎样设计才不会后面反复出交互问题？

预期第一步：先完成以下正向合同，不等用户遭遇打断、错链或滚动异常：

| 边界 | 必须先确定 |
| --- | --- |
| 启动阶段 | HTML 首帧、应用加载界面、首页和只在真实故障时出现的降级结果 |
| 媒体职责 | 每类文件、`playbackRate`、`defaultPlaybackRate`、`muted`、声音归属、预载、完成策略和失败回退 |
| 页面会话 | 首次完整加载、SPA 返回、回到顶部、整页刷新和失败重试分别如何处理剧情、欢迎动作与滚动位置 |
| Gallery | 完整目录、会话样本、当前形态、滚轮与按钮/键盘/侧卡的访问范围、向上可见性播放、`ended` 后的同边界推进 |
| 资源时序 | 冷启动预算、开门受理后的目标人物与可见海报预热、真实请求和解码证据、暂停与激活起点、失败海报 |
| 独占剧情 | 从请求受理到真实 `ended` 和导航稳定的锁；所有播放和导航入口共用查询边界 |
| 交互几何 | 固定导航与媒体边界、链接真实范围、弹层遮罩、底层排除、自定义光标定位层与视觉层 |
| 真实验收 | 刷新、返回顶部、竞争输入、样本外形态、坐标点击、音频传输和光标偏移的可观察条件 |

预期输出必须先给合同、所有者和保证，再给实现结构；具体速度、抽样数量、BGM 默认值和光标偏移只采用用户或项目值，不升级成所有角色站的固定常数。

停止条件：用户在编码前能够看懂启动时依次看到什么、各状态何时建立和重置、资源何时为哪个形态请求、各入口能访问什么、片尾如何推进、哪些动作可以打断、哪些区域可以点击，以及最终怎样用真实浏览器证明；合同中没有两个状态真源或“实现后再调”的关键边界。

## 维护检查标准

- 两条核心成功请求都先判断基线可信度并还原用户最新合同，再定位唯一生产者和消费者，不从过时 Git、局部阈值或事件补丁开始。
- 默认规则位于 `SKILL.md`；导演与会话状态位于 `interaction-director.md`，滚动成本位于 `scroll-performance.md`，真实结果位于 `browser-acceptance.md`。
- 只在片段开始后才锁定、随机样本等于完整目录、一个声音开关修改全部视频、滚动方向直接暂停 Gallery、空预热清单冒充 ready、过期异步清理修改当前节点、正常启动闪故障页、透明遮罩允许穿透等旧路径已经不存在。
- 代表性请求使用项目的具体数值通过；事前请求把这些值保留为合同参数，不错误泛化。
- 正向样本只证明默认成功路径和最小路由边界，不继续保存已经被主流程吸收的逐条纠偏故事。
