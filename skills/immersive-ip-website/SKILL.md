# Immersive IP Website

## 负责什么

把“角色图片和几段视频”组织成一个能持续表演、会回应用户、可以正常浏览的完整网站。既可用于从零规划，也可用于改造已有项目。

默认交付包含：

1. 保留范围、修改范围与体验目标。
2. 角色锚点、页面风格 DNA 和有明确用途的素材计划。
3. 完整可独立使用的图片与视频提示词包，或页面、媒体和动画导演实现。
4. 视频切换、滚动剧情与降级方案。
5. 功能、视觉和运行环境的真实浏览器验收结果。

如果用户只要求方案或提示词，停在对应阶段，不擅自修改项目。如果用户要求制作或修复网站，完成实现并沿真实用户链路验收。

## 不负责什么

- 普通静态企业站、博客或不以 IP 角色演出为核心的网站。
- 与网站无关的单张图片提示词、单段视频提示词或普通视频剪辑。
- 纯 3D 模型制作、Live2D 绑定或游戏角色控制器；除非它们只是该网站的一种素材来源。
- 未经用户授权替换角色设定、重做品牌定位或发布网站。

## 核心原则

1. **先冻结已经满意的结果。** 技术迁移不等于视觉改版。已有网站先列出保留、修改和待确认区域，并保存加载态、桌面态、移动态和关键互动截图；未获授权的区域按视觉和行为一致迁移。
2. **先定义用户会经历什么，再决定使用什么特效。** 参考网站要拆成自动表演、偶然发现、明确操作、章节过渡和返回回应，不能只模仿画面。
3. **角色设定高于生成场景。** 脸、发型、服装和原设配饰来自角色真源；锚点图里临时出现的门、钥匙、月亮或其它道具仍是场景变量，除非用户确认，不得升级为角色身份、世界观核心或全站母题。
4. **每份素材先有用途，再有提示词。** 先说明它服务哪个章节、用户会看到什么、位于哪一层、静态还是动态、是否需要拆分；说不清用途的素材不生成。
5. **生成器的真实能力决定素材方案。** 已确认的参考图数量、首尾帧能力、比例和时长选项是当前合同；不得提出生成器不支持的时长或要求用户反复纠正同一限制。
6. **提示词必须独立完整且精准。** 每条提示词包含必要的角色身份、构图或动作、背景、输出要求和排除项，不让用户拼接公共段落，不用生成器无法访问的本地简称代替角色说明。
7. **一个角色锚点、一个媒体清单、一个动画导演。** 锚点负责初始画面和切换兜底；媒体清单是文件和行为参数的唯一来源；所有播放请求由同一个导演处理。
8. **角色应该自己活着。** 待机和部分动作自动发生；空白区域、角色区域、滚动、离开后返回等正常行为可以触发惊喜。按钮是明确入口，不是全部互动。
9. **视频成片才是真实边界。** 提示词不能证明动作发生在第几秒，也不能证明尾帧可用。必须检查真实文件，再决定裁剪点、转场和滚动时机。
10. **切换方法服从素材差异。** 画面不一致时，默认使用全舞台失焦锚点转场；不要直接叠加两个人物，也不要用没有覆盖切点的扫描、烟雾或闪白装饰。
11. **叙事动画按媒体事件完成。** 会改变页面状态的动画必须等待当前视频真实 `ended`；固定计时只可作为错误超时，不能冒充播放完成。
12. **功能通过不等于设计通过。** 同时验证素材生产、文件传输、浏览器解码、导演消费、视觉保护区和用户可见结果；验收结束后关闭本次启动的服务和调试进程，确认没有占用用户端口。

## 开始前先判断

按顺序获取或检查以下信息；本地文件和代码中能找到的不要重复问用户：

- 用户已经满意、明确要求保持不变的页面、状态和交互。
- 角色设定图、身份特征、不可改变的服装和配饰。
- 参考网站或参考视频，以及用户真正喜欢的互动时刻。
- 图片和视频生成器的限制：首尾帧、时长、参考图、比例、分辨率、内容限制。
- 已生成素材的路径、分辨率、帧率、时长和可用片段。
- 目标设备、浏览器、技术栈和部署方式。
- 用户只要规划、提示词、素材处理，还是要完成网站。

然后归类：

- **从零制作**：先完成体验合同和素材计划，再写代码。
- **已有素材**：先检查真实视频，按可用内容设计互动，不根据文件名猜动作。
- **已有网站迁移或改版**：先冻结已满意区域和基准截图，再找到媒体清单、状态管理、事件入口和最终画面之间的真实链路；先完成等价迁移，之后只改授权区域。
- **只有概念**：使用 `references/project-brief-template.md` 收口目标，不急着生成大量素材。

## 主工作流

### 0. 冻结已有成果与变更边界

已有网站先建立变更表：

| 对象 | 状态 | 基准证据 | 允许动作 | 通过标准 |
| --- | --- | --- | --- | --- |
| 加载动画 | 保留 / 修改 / 待确认 | 截图、录屏、代码位置 | 原样迁移或按授权修改 | 关键帧、文案、时长和退场一致 |
| 首屏桌面版 | 保留 / 修改 / 待确认 | 固定视口截图 | | 角色裁切、标题、遮罩和控件一致 |
| 首屏移动版 | 保留 / 修改 / 待确认 | 固定视口截图 | | 功能与构图均未被擅自简化 |
| 下方章节 | 保留 / 修改 / 待确认 | 截图或用户描述 | | 只在授权范围内重新设计 |
| 互动行为 | 保留 / 修改 / 待确认 | 真实操作录屏或事件说明 | | 原用户链路仍成立 |

把用户纠偏写回同一份合同，并同步检查后续文案、提示词、素材和代码。用户否定一个元素后，不能让它以旧提示词、旧组件或旧文案继续出现。

需要迁移 React、Vue 或其它技术栈时，先说明迁移解决的是组件、状态和维护问题，不承诺框架本身会提升美术效果。等价迁移通过后，再开始授权区域的设计改造。

技术选择按真实复杂度决定：

- 多组媒体、弹层、声音、导航、章节状态和后续内容会持续增加时，可用 React 或同类组件框架统一状态与边界。
- 滚动固定、时间线和镜头进度交给 GSAP、ScrollTrigger 或等价动画工具，不把美术效果归功于组件框架。
- 简单静态页面没有状态扩展需求时，不为“更酷”强行迁移。
- Three.js、通用 UI 组件库和新的渲染层只有在具体交互或三维内容确实需要时才引入。

### 1. 写体验合同

先用一句话说明用户打开网站后会经历什么，例如：

> 角色在首屏自主待机；用户偶然点击角色或空白处会得到回应；第一次向下滚动会触发完整的章节动画，播放结束后才进入下一部分。

再列互动地图，每项至少包含：

| 触发 | 角色回应 | 页面变化 | 是否可打断 | 完成后去向 |
| --- | --- | --- | --- | --- |
| 页面加载 | 主待机 | 无 | 可 | 次待机 |
| 自动时间线 | 次待机或随机动作 | 无 | 可 | 主待机 |
| 空白或角色点击 | 直接回应 | 无 | 可 | 主待机 |
| 第一次向下滚动 | 章节动作 | 锁定首屏 | 不可 | 下一章节 |
| 返回首屏 | 欢迎回应 | 无 | 可 | 自动表演 |
| 明确按钮 | 指定动作 | 视按钮功能而定 | 按优先级 | 自动表演或功能面板 |

用户体验里没有明确价值的动画不进入素材清单。

### 2. 建立视觉锚点

锚点图不是动画尾帧合同，而是网站的视觉真源：

- 固定角色身份：脸、发型、耳朵、服装、核心配饰和标志性色彩。
- 固定首页构图：角色位置、视线、镜头高度、主体大小和界面安全区。
- 固定环境语言：材质、光线、景深、色彩比例和童话、科幻或品牌氛围。
- 为导航、标题和互动控件留出可读区域。
- 输出网站实际比例，通常先做 16:9，再为移动端确定新的 `object-position` 或独立裁切。

同时提取可跨章节复用的风格 DNA：色彩比例、材质、字体气质、线条、留白、前中后景和运动节奏。下方章节应共享这些关系，不必重复首屏场景。为环境只保留一个名称和一段清楚定义，避免把同一背景不断改名成新的世界观地点。

做一次“场景元素升级检查”：锚点中新出现但原设没有的道具、图案和建筑，默认只属于该画面。只有用户明确认可，才进入角色说明、章节标题和后续素材提示词。

需要生成锚点图、视频提示词或处理生成器限制时，完整读取 `references/creative-and-media-workflow.md`。

### 3. 规划最小视频集合

先建立素材用途表：

| 素材 | 页面目的 | 用户会看到什么 | 页面位置或图层 | 静态 / 视频 | 输入真源 | 后续处理 |
| --- | --- | --- | --- | --- | --- | --- |
| | | | | | | |

每项都能回答“为什么需要它、没有它会缺什么”后，才进入生成。需要生成多项素材时，一次性交付完整提示词包；每条都写明使用哪张输入图、生成什么、供哪个页面或视频使用。

默认从少量职责清楚的片段开始：

- 1 段主待机：呼吸、眨眼、轻微观察环境。
- 1 段次待机：操作角色标志物或改变注意力。
- 2 至 4 段动作：直接回应、消失或变形、章节过渡、靠近或邀请。

“两个待机＋四个动作”是常用起点，不是固定数量。每段只承担一个主要动作，记录：

- 触发场景。
- 动作目标。
- 生成时长。
- 真实可用起点和终点。
- 播放完后的状态。
- 是否会推动页面进入下一章节。

生成器能力表中已经确认的时长和输入方式必须原样沿用。例如工具只有两个时长选项时，只能从这两个值中选择，不能凭经验增加第三个数值。

表现性格、转身、靠近、回头等动态内容时，优先设计视频首帧和动作视频；需要静态画面时，再从合格成片中选帧。角色设定、服装细节和项目装饰等可拆分内容，适合在一张宽幅纯色绿幕母图中分区生成，之后抠图拆件。具体方法见 `references/creative-and-media-workflow.md`。

生成器不能保证尾帧时：所有片段可从同一锚点开始，但不要求回到同一画面；根据真实片段裁掉失控尾部，再由网页转场吸收画面差异。

### 4. 检查并处理真实素材

在编码前逐个检查：

- 人物身份、服装和配饰有没有漂移。
- 动作是否真的发生，发生在第几秒。
- 是否出现裸露、身体结构异常、额外人物、风格突变或错误的替代角色。
- 开头是否适合作为切入点，尾部从哪里开始失控。
- 镜头位移是否会影响网页文字和控件。
- 桌面和移动端裁切后，头部、脸和关键动作是否仍可见。

图片素材也按真实像素验收，不把棋盘格图案当作透明通道。绿幕母图逐项分类：

- **正式使用**：身份、构图和边缘均合格。
- **拆分利用**：整张不合格，但人物、边框、饰品或局部可独立使用。
- **重新生成**：核心人物、动作或主要构图错误。
- **不使用**：会破坏角色身份或页面风格，且没有可拆分价值。

只保留可用时间段，再统一为网页媒体规格。媒体文件名使用职责名，不使用生成批次名。至少准备：

```text
anchor.webp
idle-main.mp4
idle-secondary.mp4
action-response.mp4
action-story.mp4
```

根据项目继续增加动作。编码后重新检查时长、尺寸、帧率、编码格式、文件大小和浏览器播放。

### 5. 建立页面媒体层

首屏媒体舞台至少包含：

```html
<div class="hero-media">
  <img class="hero-anchor" src="anchor.webp" alt="" />
  <video class="hero-video" data-player="a" playsinline muted preload="auto"></video>
  <video class="hero-video" data-player="b" playsinline muted preload="auto"></video>
  <div class="hero-shade" aria-hidden="true"></div>
</div>
```

要求：

- 锚点始终位于视频层下方，加载失败或切换峰值时仍有完整画面。
- 两个视频层尺寸、裁切和变换完全一致。
- 首屏媒体避开固定导航栏；用明确的媒体舞台边界解决遮头，不靠随意缩小人物。
- 文案和控件位于独立层，不随视频失焦。
- 桌面和移动端分别设置角色焦点，不能只依赖 `object-fit: cover`。
- 预加载锚点和首个待机；其余片段由导演按下一步计划预加载。

### 6. 建立媒体清单和动画导演

媒体清单是所有视频参数的唯一来源：

```js
const clips = {
  idleMain: {
    src: "assets/media/idle-main.mp4",
    kind: "idle",
    next: "idleSecondary",
    switchLead: 0.45,
  },
  story: {
    src: "assets/media/action-story.mp4",
    kind: "story",
    holdAtEnd: true,
  },
};
```

动画导演统一拥有并更新：

- 当前片段和当前播放器。
- 备用播放器和预加载任务。
- 是否正在切换。
- 当前请求优先级。
- 是否锁定到片尾。
- 当前章节、首屏可见性和页面可见性。
- 动作完成后的去向。

所有自动时间线、按钮、空白点击、滚动和返回回应只向导演发请求，不直接修改 `<video>`、CSS 类或计时器。

实现导演、双缓冲、转场、事件优先级或滚动剧情时，完整读取 `references/interaction-director.md`。

### 7. 安排自主表演和互动

默认分四类：

1. **自主表演**：主待机 → 次待机 → 随机动作 → 主待机。随机动作不要连续重复。
2. **偶然发现**：点击角色画面或真正空白区域触发直接回应；链接、按钮、输入框、文案区和控制区必须排除。
3. **叙事互动**：第一次有效滚动或上滑触发章节动画，播放期间接管滚动，结束后进入目标章节。
4. **明确操作**：动画按钮、随机播放、对话入口等保留为可发现入口，并可在以后换成业务功能。

互动不是“给每个动画找一个点击事件”。先判断用户正常浏览时会做什么，再让角色回应这些行为。

### 8. 使用失焦锚点转场

当各视频人物位置、姿态或尾帧不一致时，默认流程：

1. 备用播放器完成 `loadeddata`，定位到 `0s`。
2. 旧视频继续播放，整个媒体舞台在约 150–220ms 内失焦。
3. 模糊峰值时暂停并隐藏旧视频，锚点自然露出。
4. 新视频成功 `play()` 后显示新层。
5. 媒体舞台在约 220–320ms 内重新对焦。
6. 新视频稳定后才更新当前状态和预加载计划。

新视频播放失败时恢复旧视频或锚点，并回滚导演状态。不能留下黑屏、两个可见人物或永久模糊。

只有素材首尾真正一致时才考虑直接切换或短交叉淡化。画面不一致时，禁止半透明叠加两个人物来制造“无缝”。

### 9. 让章节动画真正播完

第一次向下滚动的标准流程：

1. 确认用户仍在首屏、没有弹窗、导演已经启动。
2. 捕获有效向下滚轮或上滑手势并 `preventDefault()`。
3. 等当前切换完成，再以最高优先级播放章节视频。
4. 设置 `holdAtEnd`，阻止临近片尾时自动切回待机。
5. 锁定后续滚轮和触摸移动。
6. 等当前播放器真实触发 `ended`，并确认当前片段仍是目标片段。
7. 再滚动到目标章节并解除锁定。

错误超时只能进入失败处理：恢复滚动、保留首屏并提供重试。不能在超时后假定动画已经完成并跳页。

### 10. 处理浏览器现实条件

- 视频默认 `muted`、`playsinline`，声音必须由用户操作开启。
- 页面隐藏或首屏离开视口时暂停活动视频；返回时恢复或触发一次欢迎回应。
- `prefers-reduced-motion` 下保留静态锚点和基本导航，不强制播放角色视频。
- 触摸设备关闭依赖精确指针的磁吸、倾斜和自定义光标。
- 静态服务器或 CDN 应支持 MP4 Range 请求；媒体使用合理缓存。
- 加载失败要显示锚点和可用导航，不能让启动页永久遮挡网站。

### 11. 按真实用户链路验收

网站实现或修复后，完整读取 `references/browser-acceptance.md` 并逐项执行。最低要求：

- 首个待机确实启动，自动序列确实发生。
- 切换期间没有明显静态停顿、黑帧、双人重影或未覆盖的硬切。
- 空白或角色点击会回应，按钮和文字不会误触发该回应。
- 叙事动画播放完整后才滚动，锁定期间连续滚轮也不会提前离开。
- 离开后返回首屏会恢复有效画面和正确状态。
- 浏览器实际读取真实媒体文件，网络、控制台和媒体事件没有核心错误。
- 桌面、移动端和减少动态效果模式均可用。
- 被标记为“保留”的加载态、桌面首屏、移动首屏和关键控件与基准截图一致；不能用功能测试代替视觉对照。
- 透明图片或视频在真实页面背景上正确合成，不以文件扩展名、棋盘格或元数据标签代替实际 Alpha 检查。
- 外部入口使用真实链接；没有地址时明确标记待接入，不编造链接。
- 验收结束后关闭本次启动的开发服务器和浏览器调试进程，并确认默认端口可由用户正常启动。

## 默认输出顺序

用户只要方案时：

1. 当前限制、保留边界与可用素材。
2. 一句话体验合同和各章节目的。
3. 有明确用途的素材清单与生成顺序。
4. 每项素材完整、可独立使用的提示词。
5. 互动地图。
6. 页面和动画导演架构。
7. 实施顺序与验收标准。

用户要求制作或修复时：

1. 先检查项目结构、工作区状态、真实媒体和用户要求保留的页面状态。
2. 为保护区建立固定视口截图和行为基准。
3. 找到唯一媒体清单和唯一动画状态边界。
4. 一次性迁移所有播放入口，移除旧切换逻辑；不得保留两套导演、旧 helper 或临时恢复逻辑。
5. 先验证保护区等价迁移，再实现授权区域。
6. 启动真实网站，走完功能与视觉链路。
7. 停止本次启动的服务，确认端口释放，再报告修改文件、可观察行为、实际校验和未验证项。
