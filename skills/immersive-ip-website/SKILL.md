# Immersive IP Website

## 目标

把角色素材变成一个页面目标明确、视觉关系成立、互动结果可理解、在目标设备上真实可用的网站。设计从用户要让访问者看见、理解或完成什么开始，不从现有文案、角色标签、惯用场景或某种固定交互开始。

如果用户只要求讨论、评审、候选方向或提示词，停在相应阶段，不擅自生成图片、修改项目或继续实现。用户要求制作或修复时，完成授权范围内的实现，并沿真实素材到真实页面的链路验收。

## 边界

- 角色设定图只提供画面中能确认的身份事实，例如脸、发型、身体特征、服装、材质和配饰。它不自动提供性格、动作、场景、职业、剧情或世界观。
- 旧网页文案、文件名、历史提示词和生成模型补出的元素都不是角色事实，除非用户明确确认。
- 参考站只约束用户点名借用的部分。互动、构图、色彩、字体、文案和品牌语言分别记录，不整套复制。
- 素材是否继续用于生产页面、是否只作参考、是否替换、归档或删除，分别记录。用户要求“不再使用”时先移除所有有效引用；这不自动授权删除实体文件。
- 不把某次成功或失败的动作、环境、版式、转场参数、媒体数量或页面顺序保存成下一次任务的默认答案。
- 未经授权不替换角色设定、重做品牌定位、删除素材、安装依赖、打包、发布或推送网站。

## 工作顺序

### 1. 判断请求和授权

先区分本轮属于：

- 目标澄清或设计讨论；
- 候选视觉方向；
- 图片或视频提示词；
- 真实素材处理；
- 网站实现或修复；
- 只读验收。

“你觉得怎样”“先看看方案”不等于允许修改或生成。用户明确要求直接实现，且关键决定已有真源时，不重复索要确认。

同时锁定变更范围。用户用某些对象举例并明确要求检查同类问题时，例子是审计入口，不是完整清单；用户说只改指定内容或其它内容不动时，未点名区域就是保留基线。用户提供精确替换文案时，文字、标点、大小写和有意义的换行共同构成真源，除非用户另行授权润色。文案语气是独立决定，不因网站属于角色、奇幻或沉浸式主题就自动写成抒情设定。只要求列出、比较或审核文案时，不先替用户改写；要求先重写供审核时并列原文与候选稿，确认前不写入项目。

### 2. 建立当前事实

优先检查用户最新说明、当前运行页面、用户提供的设定图、截图、视频和实际容器。Git、旧报告、旧测试和旧文案只有在来源与时间能对应当前基线时才参与判断。

把输入分为四类：

| 类别 | 可以决定什么 | 不能越界决定什么 |
| --- | --- | --- |
| 角色事实 | 身份、外观、服装、原设配饰 | 动作、神态、环境、叙事 |
| 网站真源 | 配色关系、字体、材质、线条、已有布局与交互 | 角色设定和未确认场景 |
| 媒体事实 | 实际画幅、主体位置、动作、时长、可用区间、编码 | 文件名暗示的剧情和预期尾帧 |
| 用户决定 | 页面目的、保留边界、审美偏好、动作与环境授权 | 未表达的扩展设定 |

出现冲突时采用用户最新明确决定，并同步到设计合同、素材计划、实现和验收。不得让被否定内容通过旧提示词、旧组件或旧测试继续生效。

先判断异常由谁控制：网页 DOM、样式、媒体、favicon 和站点元数据属于项目边界；浏览器标签的布局和关闭按钮、浏览器原生控件、扩展注入层与操作系统浮层属于外部边界。只有项目边界内的问题进入网站实现。外部控制层优先给出对应设置或例外规则，不通过伪装媒体节点、破坏语义或添加脆弱 CSS 来隐藏。

截图、海报和视频画面本身也可能包含旧导航、旧文案或已退役设计。它们既是媒体，也是用户最终能看到的界面证据，不能只搜索 DOM 和源码后就认定旧内容已经消失。

### 3. 先判断页面是否需要，再定义页面目标

先盘点每个现有或候选页面：

| 页面 | 访问者能得到的独立信息或行动 | 与其它页面是否重复 | 现有内容证据 | 决定 |
| --- | --- | --- | --- | --- |
| | | | | 保留 / 合并 / 改名 / 重做 / 移除 / 待确认 |

一个页面只有在承担独立信息、探索方式、内容证据或行动入口时才需要存在。页面没有独立结果、与其它页面职责重叠且没有可靠内容来源时，优先提出合并或移除，不为了导航数量、视觉对称或沿用旧结构强行重做。用户只问“是否需要”时先给判断和依据，不把讨论自动升级为修改。

在讨论视觉形式前，用普通语言写清：

```text
页面需要完整展示什么：
访问者进入后第一眼看什么：
访问者可以怎样探索：
探索后能得到什么新信息：
页面与全站靠哪些视觉关系保持一致：
必须保留和允许改变的内容：
```

目标必须描述用户可观察结果，不能用角色标签、氛围形容词或未确认文案代替。

### 4. 先沟通设计计划，再制作

涉及明显审美判断、素材生成或较大版式变化时，先向用户说明：

1. 当前页面目标和已确认事实；
2. 素材与技术限制；
3. 准备比较的设计变量；
4. 每个候选方向会怎样改变构图、互动和媒体生产；
5. 用户确认后将交付什么。

如果方向尚未确定，提供二到三个真正不同、可比较的候选方向。区别应来自空间关系、主体与环境的结合方式、信息层级或互动机制，不是换一组形容词。解释到足以让用户判断即可，不用先生成图。

用户明确选定方向或授权 AI 自行选择后，再进入提示词或实现。

用户选定某份候选稿、参考图或生成图后，先记录其中被接受的几何、比例、负空间、层级、色彩和文字关系；它们随即成为实现合同，而不是仅供“同风格再创作”的灵感。把位图转成 SVG、CSS 或组件时先忠实还原这些关系，再做用户点名的局部调整。用户只要求缩短、移动或替换一个局部时，只改变该变量并重新核对整体，不顺手重画其它部分。

### 5. 写视觉与构图合同

进入图片、视频或页面制作前，填写：

| 决定 | 唯一真源 | 当前结论 |
| --- | --- | --- |
| 人物身份 | 角色设定图与用户确认 | |
| 动作和神态 | 动作参考或用户创作授权 | |
| 环境与光线 | 场景参考或用户创作授权 | |
| 网站配色与材质 | 当前网站或设计系统 | |
| 人物位置与占比 | 目标页面和视口 | |
| 完整可见范围 | 页面目标 | |
| 文字、导航和互动安全区 | 真实界面 | |
| 桌面与移动裁切 | 目标视口 | |

标志、图标或其它品牌识别还要记录规范母版、允许派生的尺寸版本、不得改变的轮廓与负空间，以及页眉、favicon、应用清单、加载界面、分享图和烘焙媒体等可见消费者。能使用同一矢量源时直接复用；确需派生时从同一母版生成并显式登记。

动作、环境或构图仍缺少会改变结果的决定时，停在对应空缺；只有用户授权创作时才提出候选。需要忠实落实选定视觉、把参考转成网页图形、编写提示词或处理媒体时，完整读取 `references/creative-and-media-workflow.md`。

### 6. 让媒体构图与界面共同适配

不要把响应式问题简化成 `contain` 或 `cover` 的选择。先确定主体、场景负空间和界面各自承担什么：

- 如果人物与环境的空间关系是画面价值的一部分，保留完整场景媒体，并让排版顺着原始构图组织信息和互动。
- 如果页面需要大范围改变人物位置、背景结构或内容密度，优先使用可分层素材，避免把所有关系烘焙进一段视频。
- 如果原媒体在某种宽高比下出现多余空间，先判断它是有意留白还是意外空区。有意留白应承担焦点、信息或互动职责；意外空区通过页面构图、分区尺度、受控裁切或替代媒体解决。
- 不为了填满屏幕裁掉任务要求完整展示的主体，也不保留没有视觉或交互职责的空栏。
- 桌面、超宽屏、窄屏和移动端可以采用不同布局关系，不要求同一画面机械缩放。

### 7. 先定义互动结果，再选择机制

每项互动写清：

| 输入 | 取样对象 | 反馈出现位置 | 用户得到的信息 | 触摸或键盘替代 | 结束状态 |
| --- | --- | --- | --- | --- | --- |
| | | | | | |

指针位置、取样位置和反馈位置可以相同，也可以分离；选择取决于页面目标。局部细节观察、固定观察窗、悬停响应、滚动演出和视频自主播放都是可选机制，不是角色页的默认组成。

含有推入、弹出、放大、跟随或悬浮定位的互动还要分别定义：

```text
最终布局锚点：动画结束后元素相对谁定位
运动层：从哪里进入、改变哪些视觉属性
命中层：什么区域接收点击、滚轮或关闭操作
```

最终布局由稳定的外层或布局规则负责，运动变换由内层负责，命中范围跟随可见结果。不要让同一个 `transform`、偏移量或动画库同时承担居中定位和入场位移；验收必须同时检查进入方向、运动中状态和精确终点。

只有需要多段媒体竞争、预热、切换或不可打断流程时，才建立统一媒体导演；只有需要章内滚动演出时，才建立章节生命周期。相关实现读取 `references/interaction-director.md`，滚动或多媒体性能读取 `references/scroll-performance.md`。

### 8. 以真实媒体决定网页方案

编码前检查实际文件，而不是根据提示词推断结果：

- 分辨率、比例、帧率、时长、音轨和编码；
- 人物实际位置、完整轮廓和关键动作；
- 可用起止时间、镜头漂移和尾部变化；
- 循环边界是否能直接衔接；
- 桌面与移动裁切后的信息损失；
- 是否需要 WebM、MP4、海报或透明回退。

循环策略从真实动作和边界帧选择。直接切、往返播放、动作匹配转场或短暂遮蔽都只是候选；任何产生双重轮廓、重影、跳变或动作反向违和的方案都不能通过。

### 9. 把加载时序建在真实消费者上

预加载必须服务将要显示的真实媒体节点和当前选择状态：

```text
触发
→ 正式媒体清单选择目标
→ 真实请求发出
→ 目标资源达到可解码状态
→ 页面到达
→ 同一个消费者显示目标
```

临时创建但不会成为最终播放器的元素、测试专用 URL、空任务列表或消费端手写 `ready` 都不能证明预加载成功。视频首个可解码画面出现前持续显示与页面构图一致的 poster 或静态锚点，避免页面先空白再出现视频。

预热、进入视口和成为当前活动章节是三个可能不同的时刻。即将出现的最终图片或播放器必须在预热阶段挂上目标源并完成解码，播放、入场动画和交互再按活动状态启动；不能出现后台已经下载，但最终 `<img>`、`<video>` 仍没有 `src`，直到内容进入视口后才挂载的断链。图片不能只凭 `complete` 判断成功，还要核对目标 `src` 或 `currentSrc`、非零自然尺寸和 `decode()`；视频要核对具体节点、目标 `currentSrc` 与可解码状态。

### 10. 定义页面身份与导航生命周期

页面的稳定身份、显示名称、顺序、渲染入口、导航入口、媒体预热和历史记录目标从同一份有序页面注册或等价真源产生。显示名称可以改变，但不能让旧 ID、旧标签和旧渲染分支同时继续作为有效身份。

分别写清以下生命周期的起点和恢复行为：

| 生命周期 | 页面与滚动起点 | 媒体和互动状态 | 唯一所有者 |
| --- | --- | --- | --- |
| 完整页面加载或刷新 | | | |
| 站内章节导航 | | | |
| 浏览器前进 / 后退 | | | |
| 浏览器页面缓存恢复 | | | |

如果合同要求完整加载始终从开头开始，滚动恢复必须在首次绘制前由最早的启动边界控制，应用初始化不能随后根据旧 URL、旧章节或旧存储再次跳回；同一会话内的前进后退是否恢复位置则单独决定。不能用一个“记忆页面”布尔值同时代表这些生命周期。

页面改名、合并或移除前建立迁移表，覆盖页面注册、渲染分支、导航、URL 与历史、站内入口、媒体清单与预热、样式、文档、测试以及带有旧界面的截图或视频。被移除页面原来承担的有效行动必须明确迁移或有意取消，不能留下死按钮，也不能为了保留入口继续隐藏旧路由。

### 11. 实现唯一真实边界

根据项目复杂度建立且只建立需要的单一来源：

- 页面顺序与导航目标；
- 视觉与素材合同；
- 媒体清单；
- 当前媒体或目录选择；
- 互动命中几何；
- 连续滚动进度；
- 资源预热状态。

跨页面品牌标志、站点图标和其它全局视觉资产也要有规范母版与消费者清单。页眉、favicon、应用图标、加载界面和分享元数据需要同步时，一次性迁移；对 favicon 等强缓存入口更新指纹、文件名或版本参数，并在浏览器实际读取的新地址上验收。

修复时沿“生产者 → 状态或传输边界 → 消费者 → 用户可见结果”确定根因，一次性迁移所有调用点。原功能完整迁移后移除旧监听器、旧 helper、旧状态、旧恢复路径和旧导出，不保留兼容层或半迁移状态。

### 12. 按真实用户链路验收

实现、修复或只读验收时，完整读取 `references/browser-acceptance.md`。至少证明：

- 页面从首次绘制到媒体出现没有空白或错误中间态；
- 真实媒体请求、解码和播放由正式页面触发；
- 完整刷新、站内导航、前进后退和页面缓存恢复分别符合生命周期合同；
- 目标视口中的人物完整范围、焦点、场景关系和界面分区符合合同；
- 超宽屏、常规桌面、窄屏、移动端和减少动态效果模式有意图明确的布局；
- 指针取样与反馈位置符合互动合同，命中区域与视觉区域一致；
- 入场方向、运动状态和最终几何分别正确，终点相对合同锚点没有残留位移；
- 循环边界没有重影、双重轮廓或明显跳变；
- 预加载使用最终消费者，冷缓存到达页面时仍有可用首帧或海报；
- 用户选定的视觉稿在放大轮廓和实际显示尺寸下都保持合同中的几何、比例、缺口、负空间和色彩关系；
- 全局品牌资产的所有目标消费者读取同一规范源或已登记派生，强缓存入口实际加载新版本；
- 关键结果由真实输入经过正式生产者和消费者得到，不靠测试手写状态或 mock 核心链路；
- 页面改名、合并或移除后，旧身份不再可导航、渲染、预热或请求，渲染媒体内部也没有过期界面；
- 保留区域没有因本轮改动退化。

验收证据只覆盖它生成时的文件状态。最后一次影响页面、媒体、样式、清单或测试的修改会使相关旧证据失效，必须至少重跑受影响的真实链路；等待条件来自生产页面的可观察就绪状态，固定延时只能作为超时上限，不能作为成功条件。

## 输出

先说用户现在能观察到的结论，再说明必要依据。

- 设计讨论：页面目标、事实与限制、候选方向、差异和下一步。
- 提示词：简短来源说明，加一份围绕单一画面目标的可复制提示词。
- 实现或修复：完成结果、根因、修改边界、真实浏览器证据和未验证项。
- 只读验收：通过、未通过、实际证据和影响。

不输出工具流水账，不用未经确认的角色文案替代视觉判断，也不把当前任务的失败经过或具体参数写回本 Skill。
