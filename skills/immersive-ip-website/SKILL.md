# Immersive IP Website

## 负责什么

把“角色图片和几段视频”组织成一个能持续表演、会回应用户、可以正常浏览的完整网站。既可用于从零规划，也可用于改造已有项目。

默认交付包含：

1. 保留范围、修改范围与体验目标。
2. 角色锚点、页面风格 DNA 和有明确用途的素材计划。
3. 完整可独立使用的图片与视频提示词包，或页面、媒体和动画导演实现。
4. 视频切换、滚动剧情与降级方案。
5. 功能、视觉和运行环境的真实浏览器验收结果。

如果用户只要求审视、方案或提示词，报告观察、依据和可执行建议后停在对应阶段，不擅自修改项目。如果用户要求制作或修复网站，完成实现并沿真实用户链路验收。

## 不负责什么

- 普通静态企业站、博客或不以 IP 角色演出为核心的网站。
- 与网站无关的单张图片提示词、单段视频提示词或普通视频剪辑。
- 纯 3D 模型制作、Live2D 绑定或游戏角色控制器；除非它们只是该网站的一种素材来源。
- 未经用户授权替换角色设定、重做品牌定位、删除素材、安装依赖、打包、发布或推送网站。

## 核心原则

1. **先确认基线是否可信，再冻结已经满意的结果。** 技术迁移不等于视觉改版。用户最新决定、当前运行结果、用户提供的截图或录屏是优先基线；Git、旧报告和旧测试只有在时间、分支与来源确认对应重构前状态时才能证明旧行为。已有网站先列出保留、修改和待确认区域，并保存从 HTML 首帧到加载退场、桌面态、移动态和关键互动证据；没有可靠旧版本时改用用户逐项确认的行为合同，不从过时 Git 猜测“原功能”。
2. **先把自然语言变成精确变更合同。** 写清最终页面顺序、对象是保留、增加、移动、替换还是删除、目标位于哪个元素前后或哪条线的上下方。用户只要求新增时默认保留原内容；会改变页面归属或删除内容的歧义必须先确认。
3. **参考图按借用范围拆解。** 分开记录要借的交互机制、空间层级、运动节奏与明确不要借的配色、边框、背景和文案。只说“像图中这样轮动”时，默认借轮动关系，不把参考站的配色、卡框或品牌语言带进现有网站；已有相邻章节是默认视觉真源。
4. **每个视觉决定都要有唯一真源。** 角色设定只决定脸、发型、身体特征、服装和原设配饰；姿势来自用户确认的动作参考，环境来自用户确认的场景或氛围参考，位置与裁切来自实际网页容器。不能根据服装、配饰、形态名称或所谓“气质”推导动作、场景和世界观。模型偶然生成物未经用户确认不得升级为身份、世界观核心或全站母题。
5. **用户最新决定是当前验收边界。** 用户接受某些素材瑕疵并要求直接使用时，记录为已接受问题；用户纠正页面顺序、元素、播放速度、循环或完成方式、一次性语义、预热时机或措辞后，立即重判并一次性迁移合同、媒体清单、状态生产者、消费者、测试、文档和维护样本，移除仍强制旧行为的断言与恢复逻辑。
6. **每份素材先通过入场检查。** 素材必须服务角色身份、内容证据、构图层级或交互状态，并说明它与相邻内容的关系、移除后会损失什么；说不清用途的素材不进入页面。清理现有素材时迁移清单、DOM、样式和动画调用点；只有用户授权后才删除实体文件。
7. **后续章节不能降低信息价值。** 首屏卖角色，后续也要延续角色与风格 DNA；作品章节必须让用户看到真实项目、说明、实现证据和有效入口，不能退化成通用卡片或装饰陈列。
8. **动态来自状态推进，不来自所有元素持续运动。** 先区分滚动章节、媒体转场、交互反馈和自主表演，再决定动画；每段滚动必须用新的信息、构图或状态证明其屏数成本。
9. **先选滚动架构，再排动画。** 先把章节归为首屏媒体门、显式局部交互、章内分段演出或被动内容，再选择原生滚动加 sticky、分页吸附、固定全屏舞台或局部输入步进。互相制约的要求先说明取舍，不能靠阈值和延迟反复修补。
10. **章节生命周期是唯一真源。** 每个演出章节都经历“进入构图 → 完整就位 → 章内演出 → 终态释放”。完整就位前章内进度为 0，演出期间舞台完整，终态稳定后才释放下一页；反向默认保持最高已完成状态并跳过无反馈的已消费距离。
11. **性能合同先于动画代码。** 包含滚动驱动、固定舞台、多媒体或 Canvas 时，先指定状态生产者、连续进度所有者、输入所有者、绘制效果、活动媒体、预热时机和减少动态效果方案；出现卡顿后再沿输入、状态、布局、绘制、合成和解码分层取证。
12. **生成器的真实能力决定素材方案。** 已确认的参考图数量、首尾帧能力、比例和时长选项是当前合同；不得提出生成器不支持的选项。
13. **提示词先锁定参考，再保持简洁。** 写提示词前分别确认人物、动作、环境和网页构图的来源；一条提示词只安排一个主要动作、一个场景和一个主光关系。独立完整是指执行所需决定都在正文中，不是堆叠风格词和负面词；排除项只保留会导致身份错误或素材无法使用的边界。
14. **锚点、媒体清单和导演各有唯一职责。** 首屏锚点负责初始画面和切换兜底；每个 Gallery 形态维护自己的视频与海报。媒体清单统一记录文件、播放速度、静音、声音归属、预载、预热触发和完成策略（循环、停留、重播或 `ended` 后切换）；所有播放请求由一个导演或同一画廊状态边界处理。
15. **完整目录、会话样本和当前状态分开建模。** 随机抽取的一组形态只限制本轮滚轮或自动展示，不自动排除完整目录；按钮、键盘和侧卡能访问哪些形态由体验合同单独决定。完整页面加载、SPA 返回、滚动位置和媒体完成状态也必须分开，不用其中一个代替另一个。
16. **角色应该自己活着。** 待机和部分动作自动发生；空白区域、角色区域、滚动、离开后返回等正常行为可以触发惊喜。按钮是明确入口，不是全部互动。
17. **视频成片才是真实边界。** 提示词不能证明动作发生在第几秒，也不能证明尾帧可用。必须检查真实文件，再决定裁剪点、播放速度、转场和滚动时机。
18. **切换方法服从素材差异。** 画面不一致时，默认使用全舞台失焦锚点转场；不直接叠加两个不对齐的人物，也不用未覆盖切点的装饰冒充转场。
19. **不可打断动作从请求受理开始独占。** 叙事请求一旦被接受，从等待切换、播放、真实 `ended` 到导航稳定都由导演锁定；点击、随机动作、键盘、导航和重复滚轮统一经过同一锁定边界。固定计时只用于失败恢复，不能冒充完成。
20. **先冻结章节路由和终态，再写事件处理。** 每个入口写清起点、媒体、目标章节、对齐方式、无动画备用和运行时失败结果；DOM、导航、目录和叙事路径从同一有序章节或互动合同取得目标，并通过唯一导航边界执行。
21. **可交互几何是真实边界。** 固定导航、媒体舞台、弹层、遮罩、链接卡片和自定义光标分别拥有清楚的布局、命中与视觉层。链接只覆盖可见入口，遮罩必须拦截底层，指针定位与视觉放大分层；不能用看似正确的 DOM 嵌套代替真实坐标点击。
22. **透明素材按母版到页面的链路验收。** Alpha 标签和透明扩展名不能证明可用。必须检查高质量母版、转码输出、静态海报和真实页面背景；整幅低 Alpha 方框在有损编码前清理，同时保护真实半透明细节。
23. **功能通过不等于用户通过。** 同时验证素材生产、请求发生时点、文件传输、浏览器解码、导演消费、导航终态、命中区域、视觉保护区和用户可见结果；“最终播放成功”不能证明到达前没有空白、首帧没有闪错界面或异步等待读到的是目标媒体。验收结束后只关闭本次启动的服务和调试进程。

## 开始前先判断

按顺序获取或检查以下信息；本地文件和代码中能找到的不要重复问用户：

- 本轮是征求意见、要求提示词、处理素材，还是授权修改网站；“你觉得怎样”默认是审视，不等于授权写入。
- 当前基线来自用户最新说明、当前运行网站、截图录屏、部署环境、Git 还是旧测试；分别记录时间、来源和可信度。用户说明 Git 已过时时停止用它推断旧体验，改用当前可观察结果和用户逐项确认。
- 用户已经满意、明确要求保持不变的页面、状态和交互。
- 正常启动从 HTML 首帧、静态启动壳、应用加载界面到首屏分别显示什么；无 JavaScript 或入口加载失败时才允许出现什么降级页。故障可用性不能未经授权变成正常启动中的新增界面。
- 当前页面顺序和用户要求的最终顺序；每个对象是保留、增加、移动、替换还是删除，以及“前面、下面、中间、最底下”对应的具体 DOM 或视觉锚点。
- 参考图中明确要借的机制、层级和节奏，以及不能带入当前网站的配色、边框、背景、文案和品牌符号。
- 角色设定图、身份特征、不可改变的服装和配饰。
- 每张待生成图片本身可见的动作、道具、特效和轮廓；不得根据文件序号或想象替图片命名。
- 待生成画面的人物、动作、环境、网页位置与裁切分别来自哪份参考，各参考只借什么、不借什么。动作或环境尚未确认且用户没有明确授权创作或寻找参考时，停在缺失决定，不先写一版提示词替用户做主。
- 参考网站或参考视频，以及用户真正喜欢的互动时刻。
- 每个后续章节要证明什么、目前实际展示了哪些内容证据。
- 图片和视频生成器的限制：首尾帧、时长、参考图、比例、分辨率、内容限制。
- 已生成素材的路径、分辨率、帧率、时长和可用片段。
- 用户已经明确接受、暂时忽略或要求之后再处理的素材问题。
- 每类视频和音频的播放速度、静音、声音归属、预载、预热触发、完成策略与默认开关；不能用一个全局速度、声音布尔值或 `loop` 习惯覆盖不同媒体职责。
- 首屏剧情、欢迎回应、Gallery 随机样本和刷新位置分别按完整页面加载、SPA 返回、滚动返回还是持久存储重置；“只播放一次”必须写清是哪种生命周期。
- Gallery 的完整形态目录、本次会话样本、当前活动形态与各入口可访问范围；随机抽样数量不能自动缩小按钮和键盘的完整目录。
- 首次进入 Gallery 等媒体章节前，哪个真实交互提供预热窗口、预热的是哪一个会话选中项及哪些可见海报、请求和解码怎样证明完成、进入后从头播放还是续播。空资源清单只能表示没有预热任务，不能标记为“已准备好”。
- 每个章节属于首屏媒体门、显式局部交互、章内分段演出还是被动内容；同一章节不能同时由全局页面切换和局部轮播争夺滚轮。
- 用户对“无吸附感、页面交界不可见、章内动画播完、反向立即响应”的要求是否互相制约；先选定滚动架构和取舍，不能把互斥目标留给参数调优。
- 每个演出章节在进入、完整就位、章内演出、终态释放和反向返回时，舞台、章内进度、下一页可见性与输入归属分别是什么。
- 各章节的 `offsetHeight / innerHeight`、输入是否被接管，以及进入动画和章内动画是否共用同一进度边界。
- 固定导航与首屏媒体的几何边界、弹层与底层内容的命中边界、链接实际覆盖区域，以及自定义光标的定位层、视觉层和最大跟随偏移。
- 只要任务会新增或重构滚动驱动、固定舞台、逐帧效果、Canvas 或多个同时存在的媒体层，就在编码前读取 `references/scroll-performance.md` 并完成性能职责表。已有卡顿时再补充真实路径和状态、布局绘制、合成、媒体解码与首次资源解码证据。
- 目标设备、浏览器、技术栈和部署方式。
- 用户只要规划、提示词、素材处理，还是要完成网站。

然后归类：

- **只要审视或建议**：检查用户给出的实际页面、图片或视频，先给结论、依据和可执行修改；不写项目、不生成素材。
- **只要提示词**：先建立人物、动作、环境、网页构图的参考来源表。必要来源已确认时，先用短句说明每份参考具体借什么，再交付可直接复制的精确提示词并停下；必要来源缺失时只说明缺少哪个视觉决定，不用文字自行补出场景或动作。
- **从零制作**：先完成体验合同和素材计划，再写代码。
- **已有素材**：先检查真实视频，按可用内容设计互动，不根据文件名猜动作。
- **已有网站迁移或改版**：先冻结已满意区域和基准截图，再找到媒体清单、状态管理、事件入口和最终画面之间的真实链路；先完成等价迁移，之后只改授权区域。
- **只有概念**：使用 `references/project-brief-template.md` 收口目标，不急着生成大量素材。

## 主工作流

### 0. 冻结已有成果与变更边界

已有网站先建立变更表：

| 对象 | 当前状态与基准 | 操作 | 最终归属、顺序或空间锚点 | 通过标准 |
| --- | --- | --- | --- | --- |
| 加载动画 | 截图、录屏、代码位置 | 保留 / 增加 / 移动 / 替换 / 删除 / 待确认 | | 关键帧、文案、时长和退场一致 |
| 首屏桌面版 | 固定视口截图 | | | 角色裁切、标题、遮罩和控件一致 |
| 首屏移动版 | 固定视口截图 | | | 功能与构图均未被擅自简化 |
| 下方章节 | 截图、DOM、用户描述 | | 写成 `HOME → GALLERY → CHARACTER …` 等最终顺序 | 只在授权范围内重新设计 |
| 内容与入口 | 原链接、按钮、卡片、页脚 | | 写清属于原页面、弹层还是末章 | 新增内容不吞掉原内容 |
| 互动行为 | 真实操作录屏或事件说明 | | 写清触发锚点和结束位置 | 原用户链路仍成立 |

先给每份基线标注“可信 / 部分可信 / 不可作为旧行为证据”。Git 版本、旧截图或旧报告只有与用户所说的重构前状态对得上才进入差异判断；无法对齐时不输出“原来就是这样”的结论，而是把用户当前确认写入保护合同。

默认解释是“保留现有内容并增加或移动用户点名对象”。只有用户明确说去掉、替换或确认删除时，才进入移除路径。用户使用“在某页前面”“横线下面”“页面中间”“拉到底弹出”等描述时，把它改写成最终顺序、所属章节和相邻锚点；不同理解会改变结果时先确认，不凭感觉选择。

把用户纠偏写回同一份合同，并同步检查后续文案、提示词、素材和代码。用户否定一个元素后，不能让它以旧提示词、旧组件或旧文案继续出现。

同时维护“已接受问题”列表。用户明确说某些视频瑕疵先不用管、直接使用时，记录文件、问题和接受范围；后续验收只确认它没有引入新的功能或合成错误，不再把已接受瑕疵报告成阻塞项。

需要迁移 React、Vue 或其它技术栈时，先说明迁移解决的是组件、状态和维护问题，不承诺框架本身会提升美术效果。等价迁移通过后，再开始授权区域的设计改造。

技术选择按真实复杂度决定：

- 多组媒体、弹层、声音、导航、章节状态和后续内容会持续增加时，可用 React 或同类组件框架统一状态与边界。
- 滚动固定、时间线和镜头进度交给 GSAP、ScrollTrigger 或等价动画工具，不把美术效果归功于组件框架。
- 简单静态页面没有状态扩展需求时，不为“更酷”强行迁移。
- Three.js、通用 UI 组件库和新的渲染层只有在具体交互或三维内容确实需要时才引入。

### 1. 写体验合同

先用一句话说明用户打开网站后会经历什么，例如：

> 角色在首屏自主待机；用户偶然点击角色或空白处会得到回应；第一次向下滚动会触发完整的章节动画，播放结束后才进入下一部分。

再列互动地图，每项至少包含：

| 触发 | 角色回应 | 页面变化 | 是否可打断 | 完成后去向 |
| --- | --- | --- | --- | --- |
| 页面加载 | 主待机 | 无 | 可 | 次待机 |
| 自动时间线 | 次待机或随机动作 | 无 | 可 | 主待机 |
| 空白或角色点击 | 直接回应 | 无 | 可 | 主待机 |
| 第一次向下滚动 | 章节动作 | 锁定首屏 | 不可 | 下一章节 |
| 返回首屏 | 欢迎回应 | 无 | 可 | 自动表演 |
| 明确按钮 | 指定动作 | 视按钮功能而定 | 按优先级 | 自动表演或功能面板 |

用户体验里没有明确价值的动画不进入素材清单。

互动表里的“完成后去向”必须来自最终页面顺序或明确业务动作。表示“进入下一章节”的入口从同一有序章节真源计算目标；成功播放、体验合同明确允许直接进入的无动画降级、滚动提示按钮和减少动态效果模式复用同一目标。播放中途发生意外错误仍按失败合同留在当前可用页面并提供重试。明确写着“读取角色档案”之类允许跳章的按钮是另一项业务动作，不能和“进入下一章节”共用含糊的备用逻辑。

### 实现前硬门槛

在写滚轮监听器、ScrollTrigger、动画组件或媒体播放入口前，先为每类章节选择主架构：

| 架构 | 适用章节 | 能保证什么 | 必须接受或另行解决什么 |
| --- | --- | --- | --- |
| 原生滚动 + sticky 舞台 | 大多数章内分段演出 | 不接管连续输入，章内推进直接映射滚动距离 | 普通页面过渡可能短暂出现上下章节交界 |
| 分页吸附 | 用户明确接受离散落页 | 每次稳定落在完整页面 | 触控板和反向滚动可能有黏滞感，必须专项验收 |
| 固定全屏舞台 + 内容换场 | 同时要求无强制吸附感且换场时不出现两页拼接 | 页面切换期间始终只有一个全屏舞台 | 状态、可访问性和路由复杂度更高 |
| 局部输入步进 | Gallery 等用户明确可逐项切换的有限交互 | 一次输入推进一个离散形态或状态 | 只能在完整就位的局部区域接管；向上滚动默认立即释放 |

不要把几种架构同时叠在一章上补救手感。选定架构后，填写统一生命周期；被动内容不需要章内演出，可直接从完整就位进入释放：

| 阶段 | 舞台几何 | 章内进度 | 下一页 | 反向行为 |
| --- | --- | --- | --- | --- |
| 进入构图 | 章节尚未完整占据表演视口 | 固定为 0 | 不参与章内演出 | 原生返回 |
| 完整就位 | 舞台达到合同锚点 | 仍为 0 | 不可进入视口 | 保持当前稳定画面 |
| 章内演出 | 舞台保持完整 | 从 0 单调推进到 1 | 不可进入视口 | 保持已达到的最高进度 |
| 终态释放 | 终态已经可见且稳定 | 固定为 1 | 才允许进入视口 | 跳过视觉不再变化的已消费距离 |

再填写性能与路由职责：

| 章节或入口 | 离散状态生产者 | 连续进度唯一所有者 | 大面积绘制或合成效果 | 同时活动媒体 | 资源预热时机 | 减少动态效果 | 最终路由与稳定终态 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| | | | | | | | |

再为有到达期限的媒体填写资源时序；预热使用与最终消费者相同的会话选择和媒体清单，不复制第二份静态 URL 列表：

| 目标媒体 | 选择状态唯一真源 | 冷启动是否请求 | 预热触发 | 请求已发生证据 | 可用证据 | 到达时显示 | 激活时行为 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Gallery 首形态与相邻海报 | | | 首屏剧情受理 / 其它 | 真实 Resource Timing 或 Network | 图片 `decode()` / 视频 `readyState` 与目标 URL | 海报 / 视频首帧 | `currentTime = 0` 后播放 / 续播 |

同一张表之后补齐三份会改变实现的合同：

| 媒体职责 | 文件真源 | 播放速度 | muted | 声音归属 | 完成策略 | preload / 预热 | 失败回退 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| | | | | 人物视频 / 独立 BGM / 其它 | loop / hold / replay / advance-on-ended | | |

| 状态 | 首次完整加载 | SPA 返回 | 回到顶部 | 整页刷新 | 播放失败后的明确重试 |
| --- | --- | --- | --- | --- | --- |
| 首屏剧情 / 欢迎回应 / Gallery 样本 / 滚动位置 | | | | | |

| 交互对象 | 可见区域 | 实际命中区域 | 上层拦截层 | 底层不得触发 | 真实坐标验收 |
| --- | --- | --- | --- | --- | --- |
| 链接卡片 / 弹层 / 遮罩 / 自定义光标 | | | | | |

同时把关键验收写成实现无关的可观察条件，例如“真实 `ended` 后才开始导航”“目标按合同锚点稳定”“离散章节变化不在每个滚动帧派发全局状态”。表中存在未归属项，或成功、无动画备用和运行时失败仍共用含糊的 `next`、DOM 查询或固定计时，不进入编码；先从现有项目确认，只有不同答案会改变用户结果时才询问。

生命周期的硬验收至少包含：章节尚未完整就位时章内进度始终为 0；章内演出期间下一页不进入视口；下一页首次可见时前一章已经稳定在终态；反向第一次有效输入立即产生位移或状态反馈，不要求用户滚过一段视觉完全不变的已消费距离。

参考图同时建立借用表：

| 参考对象 | 明确借用 | 明确不借 | 当前网站的真源 |
| --- | --- | --- | --- |
| 卡片轮动参考 | 前后层级、切换方式、焦点变化 | 黑色背景、厚卡框、参考站字体与装饰 | 相邻章节的背景、线条、字号和强调色 |

参考只约束用户点名的部分。没有得到授权时，不因参考图“更完整”就替换现有网站的配色、背景、边框或内容结构。

再为后续章节建立推进合同：

| 章节 | 类型与主架构 | 要证明什么 | 主要内容证据 | 完整就位构图 | 章内演出 | 终态与反向 | 预计屏数 | 可移除项 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| | | | | | | | | |

每个状态都应带来新的信息、构图关系或明确反馈。预计屏数不是为了把页面做长，而是提前约束用户要滚动多久；移除项用于找出没有信息、结构或交互职责的素材。

### 2. 建立视觉锚点

锚点图不是动画尾帧合同，而是网站的视觉真源：

- 固定角色身份：脸、发型、耳朵、服装、核心配饰和标志性色彩。
- 固定首页构图：角色位置、视线、镜头高度、主体大小和界面安全区。
- 固定环境语言：材质、光线、景深、色彩比例和童话、科幻或品牌氛围。
- 为导航、标题和互动控件留出可读区域。
- 输出网站实际比例，通常先做 16:9，再为移动端确定新的 `object-position` 或独立裁切。

同时提取可跨章节复用的风格 DNA：色彩比例、材质、字体气质、线条、留白、前中后景和运动节奏。下方章节应共享这些关系，不必重复首屏场景。为环境只保留一个名称和一段清楚定义，避免把同一背景不断改名成新的世界观地点。

已有网站新增 GALLERY、形态轮播或透明立绘舞台时，以前后相邻章节为默认视觉真源。参考图中的“卡片轮动”不自动要求给主立绘加框；边框只有在承担选中状态、裁切窗口或交互边界时才保留。主角不能为了塞进装饰框而明显缩小，透明人物可以直接进入页面空间，由排版线、序号、局部光场和前后形态建立层级。深色角色或恶魔形态也不自动把整章改成黑色背景。

做一次“场景元素升级检查”：锚点中新出现但原设没有的道具、图案和建筑，默认只属于该画面。只有用户明确认可，才进入角色说明、章节标题和后续素材提示词。

后续章节延续的是色彩、材质、字体、线条、空间层级、角色存在方式和运动节奏，不是复制首屏场景。每章都要有自己的信息任务和主要证据；角色可以作为引导者、操作者或叙事线索，但不能遮住真实项目、过程或结果。不能因为首屏足够独特，就允许作品区回落成任意网站都能使用的模板卡片。

需要生成锚点图、视频提示词或处理生成器限制时，完整读取 `references/creative-and-media-workflow.md`。进入提示词阶段前，必须先完成其中的视觉来源表；网站截图只能提供配色、材质、空间层级、人物位置和裁切关系，不能代替动作或环境参考。

### 3. 规划最小视频集合

先建立素材用途表：

| 素材 | 页面目的 | 用户会看到什么 | 页面位置或图层 | 静态 / 视频 | 输入真源 | 后续处理 |
| --- | --- | --- | --- | --- | --- | --- |
| | | | | | | |

每项都能回答“为什么需要它、没有它会缺什么”后，才进入生成。需要生成多项素材时，一次性交付完整提示词包；每条都写明使用哪张输入图、生成什么、供哪个页面或视频使用。

默认从少量职责清楚的片段开始：

- 1 段主待机：保持页面基本生命感，动作幅度来自已确认参考和真实容器。
- 1 段次待机：只有它能提供与主待机不同的页面反馈且已有动作来源时才制作。
- 若干互动动作：每段对应一个已确定的触发、用户可见结果和动作参考。

数量由页面需要和生成器能力决定，不预设“两个待机＋四个动作”。每段只承担一个主要动作，记录：

- 触发场景。
- 动作目标。
- 生成时长。
- 真实可用起点和终点。
- 播放完后的状态。
- 是否会推动页面进入下一章节。

生成器能力表中已经确认的时长和输入方式必须原样沿用。例如工具只有两个时长选项时，只能从这两个值中选择，不能凭经验增加第三个数值。

为每个待生成形态建立逐图观察卡：该图承担的参考职责、可见身份特征、原服装、当前姿势、画面已有特效或物品，以及明确借用和不借用的内容。形态名称优先使用用户提供的名称或文件名；证据不足时用“第 2 张”等中性编号，不补写身份、性格和世界观。完整提示词和动画安全首帧要求见 `references/creative-and-media-workflow.md`。

表现转身、靠近、回头等动态内容时，优先根据已确认动作参考设计视频首帧和动作视频；需要静态画面时，再从合格成片中选帧。角色设定、服装细节和已经通过用途检查的界面元素等可拆分内容，适合在一张宽幅纯色绿幕母图中分区生成，之后抠图拆件。具体方法见 `references/creative-and-media-workflow.md`。

生成器不能保证尾帧时：所有片段可从同一锚点开始，但不要求回到同一画面；根据真实片段裁掉失控尾部，再由网页转场吸收画面差异。

### 4. 检查并处理真实素材

在编码前逐个检查：

- 人物身份、服装和配饰有没有漂移。
- 动作是否真的发生，发生在第几秒。
- 是否出现裸露、身体结构异常、额外人物、风格突变或错误的替代角色。
- 开头是否适合作为切入点，尾部从哪里开始失控。
- 镜头位移是否会影响网页文字和控件。
- 桌面和移动端裁切后，头部、脸和关键动作是否仍可见。

图片素材也按真实像素验收，不把棋盘格图案当作透明通道。绿幕母图逐项分类：

- **正式使用**：身份、构图和边缘均合格。
- **拆分利用**：整张不合格，但人物、边框或已经通过用途检查的局部可独立使用。
- **重新生成**：核心人物、动作或主要构图错误。
- **不使用**：会破坏角色身份或页面风格，且没有可拆分价值。

只保留可用时间段，再统一为网页媒体规格。媒体文件名使用职责名，不使用生成批次名。至少准备：

```text
anchor.webp
idle-main.mp4
idle-secondary.mp4
action-response.mp4
action-story.mp4
```

根据项目继续增加动作。编码后重新检查时长、尺寸、帧率、编码格式、文件大小和浏览器播放。

透明 MOV 转成网站 WebM 后若在浅色背景上出现整幅方框，先回到带 Alpha 母版，在有损编码前清理低 Alpha 噪点；不要对已经压缩的 WebM 反复转码。使用 `scripts/clean_alpha_video.py` 直接完成母版转码、Alpha 前后统计和透明海报生成。默认阈值为 8；只有页面仍有方框且发丝、薄纱、翅膀和粒子没有受损时才尝试 16。黑色半透明边缘是另一类问题，只有实际观察到黑边时才进入逐帧颜色修复，不和低 Alpha 清理默认绑定。

### 5. 建立页面媒体层

首屏媒体舞台至少包含：

```html
<div class="hero-media">
  <img class="hero-anchor" src="anchor.webp" alt="" />
  <video class="hero-video" data-player="a" playsinline muted preload="auto"></video>
  <video class="hero-video" data-player="b" playsinline muted preload="auto"></video>
  <div class="hero-shade" aria-hidden="true"></div>
</div>
```

要求：

- 锚点始终位于视频层下方，加载失败或切换峰值时仍有完整画面。
- 两个视频层尺寸、裁切和变换完全一致。
- 首屏媒体避开固定导航栏；用明确的媒体舞台边界解决遮头，不靠随意缩小人物。
- 文案和控件位于独立层，不随视频失焦。
- 桌面和移动端分别设置角色焦点，不能只依赖 `object-fit: cover`。
- 预加载锚点和首个待机；其余片段由导演按下一步计划预加载。

### 6. 建立媒体清单和动画导演

媒体清单是所有视频参数的唯一来源：

```js
const clips = {
  idleMain: {
    src: "assets/media/idle-main.mp4",
    kind: "idle",
    next: "idleSecondary",
    playbackRate: 1,
    muted: true,
    switchLead: 0.45,
  },
  story: {
    src: "assets/media/action-story.mp4",
    kind: "story",
    playbackRate: projectMediaContract.story.playbackRate,
    muted: true,
    lockUntilEnd: true,
    completion: "advance-on-ended",
  },
};
```

动画导演统一拥有并更新：

- 当前片段和当前播放器。
- 备用播放器和预加载任务。
- 是否正在切换。
- 当前请求优先级。
- 是否有已经受理但尚未完成切换的独占请求。
- 当前片段是否锁定到真实片尾与后续导航稳定。
- 当前章节、首屏可见性和页面可见性。
- 本次完整页面加载已经消费的单次剧情与随机样本。
- 动作完成后的去向。

所有自动时间线、按钮、空白点击、滚动和返回回应只向导演发请求，不直接修改 `<video>`、CSS 类或计时器。

预热只让目标资源进入浏览器并达到合同中的可用状态，不提前播放或消耗动作。媒体真正激活时重新应用清单参数，并按合同从 `0s` 播放或续播。异步切换完成前记录目标媒体身份和操作编号；回调、清理和验收都再次核对同一个节点与 URL，不能把已经被复用成当前人物的节点当作旧视频回收。

实现导演、双缓冲、转场、事件优先级或滚动剧情时，完整读取 `references/interaction-director.md`。

### 7. 安排自主表演和互动

默认分四类：

1. **自主表演**：主待机 → 次待机 → 随机动作 → 主待机。随机动作不要连续重复。
2. **偶然发现**：点击角色画面或真正空白区域触发直接回应；链接、按钮、输入框、文案区和控制区必须排除。
3. **叙事互动**：第一次有效滚动或上滑触发章节动画，播放期间接管滚动，结束后进入目标章节。
4. **明确操作**：动画按钮、随机播放、对话入口等保留为可发现入口，并可在以后换成业务功能。

互动不是“给每个动画找一个点击事件”。先判断用户正常浏览时会做什么，再让角色回应这些行为。

为每个动画标明归属：自主表演、媒体转场、滚动驱动的章节状态或即时交互反馈。短暂的透明度、颜色和状态变化主要用于切换与反馈，不是替代所有运动的统一答案；滚动章节可以有明显位移、镜头和构图变化，但这些变化必须揭示内容或推动叙事。自主循环只有在表达角色生命感或持续系统状态时才保留，不能用漂浮、视差和摆动填满空白。

### 8. 使用失焦锚点转场

当各视频人物位置、姿态或尾帧不一致时，默认流程：

1. 备用播放器完成 `loadeddata`，定位到 `0s`。
2. 旧视频继续播放，整个媒体舞台在约 150–220ms 内失焦。
3. 模糊峰值时暂停并隐藏旧视频，锚点自然露出。
4. 新视频成功 `play()` 后显示新层。
5. 媒体舞台在约 220–320ms 内重新对焦。
6. 新视频稳定后才更新当前状态和预加载计划。

新视频播放失败时恢复旧视频或锚点，并回滚导演状态。不能留下黑屏、两个可见人物或永久模糊。

只有素材首尾真正一致时才考虑直接切换或短交叉淡化。画面不一致时，禁止半透明叠加两个人物来制造“无缝”。

### 9. 让章节动画真正播完

第一次向下滚动的标准流程：

1. 确认用户仍在首屏、没有弹窗、导演已经启动。
2. 捕获有效向下滚轮或上滑手势并 `preventDefault()`。
3. 在请求受理时立即写入本次完整页面加载的“剧情已消费”状态，并建立独占锁；返回顶部不能清除它，整页刷新按合同重新建立。
4. 等导演就绪与当前切换收口，再以最高优先级请求带 `lockUntilEnd` 的章节视频；等待期间同样保持锁定。
5. 锁定滚轮、触摸、键盘、导航、随机动作、空白点击和其它播放入口；所有入口通过导演或导航控制器查询同一个锁定状态。
6. 等当前播放器真实触发 `ended`，并确认当前播放器和片段仍是请求目标。
7. 通过唯一导航边界进入互动合同中的目标章节，等待滚动稳定后再解除独占锁。

错误超时只能进入失败处理：恢复滚动、保留首屏并提供重试。不能在超时后假定动画已经完成并跳页。

“下一章节”的目标不在滚轮监听器、按钮回退和动画完成回调中分别硬编码。它来自有序章节清单或互动路由；导航函数统一处理关闭弹层、滚动方式、固定导航留白和活动章节更新。DOM 顺序正确并不能证明开门动画去向正确，必须继续检查完成回调、失败降级和用户最终落点。

首屏故事门控和后续章节生命周期必须分开：只有首屏可为等待真实媒体结束而暂时接管输入。后续普通章节由浏览器拥有输入，sticky 或固定全屏舞台只消费滚动位置；只有体验合同明确列出的局部步进可以有限接管。所有阶段、屏数预算、反向处理与验收方法见 `references/interaction-director.md`。

上面的完整媒体门流程只适用于首屏或产品明确指定的不可打断叙事，不得推广成全站强制落页。普通分段章节按统一生命周期推进：进入只负责构图，完整就位后章内进度才从 0 开始，终态稳定后才释放到下一页。连续视觉值直接映射滚动位置，不用延迟追赶的时间动画模拟滚动。

Gallery 是显式局部交互，不等同于普通章节。它分别维护完整目录、本次完整页面加载中稳定的随机样本和当前活动形态。只有 Gallery 已完整就位且用户向下推进样本中尚未看过的形态时，滚轮才消费一次输入；样本完成后立即释放给原生向下滚动。向上滚动直接返回上一章，不倒序消费样本，也不因滚动方向暂停仍在视口内的活动视频。左右按钮、方向键和侧卡按体验合同访问完整目录，并与滚轮复用同一形态选择、转场和播放边界；返回任何已加载形态时按该条目的完成与回访合同从头播放或恢复。

如果首屏剧情提供 Gallery 到达前的预热窗口，剧情请求受理后立即让 Gallery 自己当前会话选中的首形态视频、当前海报和可见相邻海报开始预热；首屏冷启动阶段仍按合同控制资源预算。预热时视频保持暂停，Gallery 激活后从 `0s` 播放。Gallery 的完成策略由媒体合同决定：需要持续轮换时由真实 `ended` 经过同一形态选择边界切换下一个人物；需要循环时必须确认素材接缝或约定转场，不能仅设置 `loop` 就宣称顺滑。

返回首屏自动播放“欢迎回来”属于明确彩蛋，是“反向默认不重演”的产品级例外。它与“每次完整页面加载只播放一次”的首屏剧情使用不同状态；欢迎动作不能重置剧情消费状态，也不能让普通 Character、Personality 或 Works 章节向上返回时重置进度。

### 10. 处理浏览器现实条件

- 视频默认 `muted`、`playsinline`。如果合同规定人物视频始终静音，声音按钮只控制独立 BGM，不能改写任一人物播放器；BGM 的默认开关、循环、音量和起止区间采用用户最新决定。
- 每类媒体从清单读取自己的 `playbackRate`，切换播放器和回访重播都重新应用；不使用一个全局速度覆盖 Hero、Gallery、Personality 和章节动作。
- 下载器或浏览器扩展会接管 `<audio src>` 中的音频直链时，由页面 `fetch` 音频并转换成内部 Blob URL 再交给播放器；保留真实请求、播放、循环与错误恢复证据，不要求用户修改全局下载器设置。
- 如果合同要求整页刷新回到首页顶部，在应用框架启动前设置 `history.scrollRestoration = "manual"` 并清零滚动；SPA 返回首页不能伪装成整页刷新，也不能重新消费一次性剧情。
- 页面隐藏或首屏离开视口时暂停活动视频；返回时恢复或触发一次欢迎回应。
- `prefers-reduced-motion` 下保留静态锚点和基本导航，不强制播放角色视频。
- 触摸设备关闭依赖精确指针的磁吸、倾斜和自定义光标。桌面自定义光标需要跟随效果时，定位外层和缩放视觉层分开，弹簧偏移设最大半径，停止后回到真实指针位置。
- 固定导航高度只由一个布局变量生产，首屏媒体从导航下沿开始；滚动状态不能再叠加额外空白。用户要求隐藏页面滚动条时同时覆盖 Firefox 与 WebKit，并保留原生滚动和键盘可访问性。
- 静态服务器或 CDN 应支持 MP4 Range 请求；媒体使用合理缓存。
- 加载失败要显示锚点和可用导航，不能让启动页永久遮挡网站。
- 正常启动的 HTML 首帧应与应用加载界面视觉连续；静态介绍或故障页默认隐藏，只在无 JavaScript、入口加载失败或明确超时后出现。既要验证故障路径可用，也要验证它从未在正常启动帧中闪现。

### 11. 按真实用户链路验收

网站实现或修复后，完整读取 `references/browser-acceptance.md` 并逐项执行。最低要求：

- 首个待机确实启动，自动序列确实发生。
- 从页面导航开始记录 HTML 首帧、应用加载界面和首屏接管；正常路径没有闪现故障降级页，不能只在应用挂载后检查 `.boot-screen`。
- 切换期间没有明显静态停顿、黑帧、双人重影或未覆盖的硬切。
- 空白或角色点击会回应，按钮和文字不会误触发该回应。
- 叙事请求受理后到导航稳定都不可被点击、随机动作、键盘、导航和连续滚轮打断；返回顶部再次向下不重播，按合同完成整页刷新后才重新可用。
- 叙事成功路径、同一入口的无动画备用路径和减少动态效果路径使用同一“下一章节”目标；动画结束并且滚动稳定后，目标章节与互动合同约定的导航锚点对齐，活动导航和用户可见内容一致。
- 离开后返回首屏会恢复有效画面和正确状态。
- 浏览器实际读取真实媒体文件，网络、控制台和媒体事件没有核心错误。
- 桌面、移动端和减少动态效果模式均可用。
- 后续章节在进入构图、完整就位、章内演出和终态释放阶段都符合信息与构图合同，没有正向无推进或反向无反馈的长距离空滚；记录桌面与移动端各章节的 `offsetHeight / innerHeight`。
- 对每个章内分段演出记录进入、完整就位、演出中段、终态释放和反向返回五个检查点：完整就位前进度为 0，演出期间下一页不可见，下一页出现前本章已在终态，反向第一次有效输入立即有反馈。
- Gallery 的完整目录、随机样本和当前活动形态互不覆盖；滚轮只消费样本，按钮、方向键和侧卡按合同访问完整目录。向上滚动立即释放且视口内活动视频继续播放，返回任意形态按合同从头播放或恢复。
- Gallery 预热在合同指定的首屏交互窗口真实发起，目标视频和海报 URL 来自当前会话选择；到达前能看到真实请求与可用证据，到达后不经过空白等待。等待条件同时绑定预期形态 ID、`currentSrc`、可见节点和播放状态，不能让旧的 outgoing 视频满足泛化的 `readyState > 0`。
- 弹层打开后，真实坐标点击弹层外部命中遮罩而不是底层作品或链接；卡片外点击不打开新页面，作品展示本身只有在合同明确要求时才是链接。自动出现的末章卡片由章节稳定终态触发，不依赖鼠标位于视口某个横向位置。
- 自定义光标快速移动时保留合同规定的弹簧偏移上限，停止后圆心回到红点；链接悬停放大不改变定位坐标。
- 固定导航下沿与首屏媒体上沿在加载、滚动和刷新后保持同一边界；隐藏滚动条不会破坏真实滚动。
- 只要实现包含滚动驱动、固定舞台、Canvas 或多媒体层，就按编码前的性能职责表检查真实输入、视觉推进和最终落点；用户反馈卡顿时再用同一路径证明主要成本位于状态、布局绘制、合成、资源解码还是媒体解码。不能等出现投诉后才第一次建立这些边界。
- 作品区可见真实项目名称、说明、实现证据和有效入口，没有被角色立绘或生成式装饰替代。
- 独立装饰均能说明角色身份、内容、结构或交互职责；删除候选没有残留在 DOM、动画调用点或素材清单中。
- 被标记为“保留”的加载态、桌面首屏、移动首屏和关键控件与基准截图一致；不能用功能测试代替视觉对照。
- 透明图片或视频在真实页面背景上正确合成，不以文件扩展名、棋盘格或元数据标签代替实际 Alpha 检查。
- 透明母版、WebM 和海报的帧数、时长与 Alpha 分布形成同一条证据链；全画布低 Alpha 方框已经消失，真实半透明细节仍在。
- 外部入口使用真实链接；没有地址时明确标记待接入，不编造链接。
- 验收结束后关闭本次启动的开发服务器和浏览器调试进程，并确认默认端口可由用户正常启动。

## 默认输出顺序

第一段先说用户现在能观察到的结论；诊断任务说根因，制作或修复任务说完成结果。后续才说明合同、实现边界和证据，不用工具过程或抽象术语开场。

用户只要图片或视频提示词时：

1. 先用简短来源表说明人物、动作、环境和网页构图分别借用哪份参考及具体范围。
2. 来源齐全后直接给一份可复制的提示词；提示词围绕一个画面目标组织，不附带长篇设计理论、提示词解释、巨大负面清单或未经要求的备选方案。
3. 来源不齐全时，准确指出缺少的视觉决定并停下。只有用户明确授权创作新场景、寻找或生成动作与环境参考后，才提出候选方向；候选方向要先让用户确认，不能自动升级为提示词真源。用户明确要求 AI 自行选择并直接交付时例外。

用户只要方案时：

1. 当前限制、准确页面顺序、保留 / 增加 / 移动 / 替换边界与可用素材。
2. 参考图明确借什么、不借什么，以及当前网站的视觉真源。
3. 一句话体验合同、各章节类型、信息任务和主要证据。
4. 滚动架构选择、要求取舍与章节生命周期。
5. 章内可见状态、预计屏数、输入边界与反向策略。
6. 编码前性能职责表、互动路由和稳定终态断言。
7. 有明确用途的素材清单、逐图证据与生成顺序。
8. 每项素材完整、可独立使用的提示词。
9. 互动地图。
10. 页面和动画导演架构。
11. 实施顺序、已接受问题与验收标准。

用户要求制作或修复时：

1. 先检查项目结构、工作区状态、真实媒体、用户要求保留的页面状态，以及每份旧行为证据的来源与可信度；旧 Git 不可靠时立即切换到用户确认和当前运行基线。
2. 写出最终页面顺序、内容归属、空间锚点、媒体策略、页面会话语义和真实命中区域，默认保留未点名移除的原内容。
3. 为保护区建立固定视口截图和行为基准，并写明参考图借用范围。
4. 沿“配置或素材生产者 → 状态与传输边界 → DOM、媒体或导航消费者 → 用户可见结果”复现问题，先确定根因属于哪个唯一边界。
5. 找到唯一媒体清单、唯一章节生命周期、唯一有序章节真源、唯一导航入口和唯一命中所有者。
6. 为每章分类并选择滚动架构，明确互相制约的体验要求、输入归属和反向策略。
7. 在编码前完成性能职责、媒体播放、页面会话、交互命中、所有入口路由和稳定终态合同。
8. 记录各章节的信息任务、进入 / 完整就位 / 章内演出 / 终态释放状态、滚动屏数和素材入场理由。
9. 一次性迁移所有消费者，移除被替代的监听器、播放入口、链接范围、旧 helper 和恢复逻辑；不保留两套导演或半迁移状态。
10. 先验证保护区等价迁移，再实现授权区域。
11. 启动真实网站，用真实导航首帧、刷新、滚轮、点击、键盘、媒体事件和网络请求走完生产者到可见结果；对有时序要求的链路记录“触发 → 请求 → 可用 → 到达 → 可见”，再检查性能、视觉、声音和透明素材链路。
12. 停止本次启动且不再需要的服务，再报告用户可观察结果、根因、修改边界、实际证据、已接受问题和未验证项。

## 脚本路由

- `scripts/clean_alpha_video.py`
  - 什么时候运行：高质量透明 MOV 或其它 Alpha 母版需要转换成网页 VP9 WebM，尤其是现有 WebM 在浅色背景上出现整幅淡色或灰黑方框时。
  - 输入与输出：输入一个不修改的 Alpha 母版，输出清理后的 `.webm`；可用 `--poster` 同步生成透明 WebP 海报，用 `--report` 保存母版与输出的真实 Alpha、四角和外圈统计。
  - 默认动作：在 8-bit Alpha 中把 `0 < alpha <= 8` 清零后再进行第一次有损编码，随后用 `libvpx-vp9` 解码成真实 Alpha 平面核对帧数和分布。脚本不会自动修复黑色边缘，也不能代替真实页面背景上的视觉验收。
  - 写入边界：脚本不修改输入文件，目标已存在时默认停止；只有用户明确允许覆盖时才传 `--overwrite`。批量素材逐个运行并保留每份报告，不用一个成功样本代替整批验收。
